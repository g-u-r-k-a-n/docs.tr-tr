## <a name="appendix"></a><span data-ttu-id="00727-233">Ek</span><span class="sxs-lookup"><span data-stu-id="00727-233">Appendix</span></span>  
 <span data-ttu-id="00727-234">Sıkı sürüm gerektiğinde genel veri sözleşmesi sürüm kılavuzu, veri sözleşmelerini değişmez olarak ele almak ve değişiklikler gerektiğinde yenilerini oluşturmaktır.</span><span class="sxs-lookup"><span data-stu-id="00727-234">The general data contract versioning guidance when strict versioning is needed is to treat data contracts as immutable and create new ones when changes are required.</span></span> <span data-ttu-id="00727-235">Her yeni veri sözleşmesi için yeni bir sınıf oluşturulması gerekir, bu nedenle eski veri sözleşmesi sınıfı açısından yazılmış varolan kodu almak ve yeni veri sözleşmesi sınıfı açısından yeniden yazmak zorunda kalmaktan kaçınmak için bir mekanizma gereklidir.</span><span class="sxs-lookup"><span data-stu-id="00727-235">A new class needs to be created for each new data contract, so a mechanism is needed to avoid having to take existing code that was written in terms of the old data contract class and rewrite it in terms of the new data contract class.</span></span>  
  
 <span data-ttu-id="00727-236">Bu mekanizmalardan biri, arabirimleri uygulayan veri sözleşmesi sınıfları yerine arabirimler açısından her veri sözleşmesinin üyelerini tanımlamak ve iç uygulama kodu yazmak için arabirimleri kullanmaktır.</span><span class="sxs-lookup"><span data-stu-id="00727-236">One such mechanism is to use interfaces to define the members of each data contract and write internal implementation code in terms of the interfaces rather than the data contract classes that implement the interfaces.</span></span> <span data-ttu-id="00727-237">Bir hizmetin sürüm 1 için `IPurchaseOrderV1` aşağıdaki kodu `PurchaseOrderV1`bir arabirim ve bir gösterir:</span><span class="sxs-lookup"><span data-stu-id="00727-237">The following code for version 1 of a service shows an `IPurchaseOrderV1` interface and a `PurchaseOrderV1`:</span></span>  
  
```csharp  
public interface IPurchaseOrderV1  
{  
    string OrderId { get; set; }  
    string CustomerId { get; set; }  
}  
  
[DataContract(  
Name = "PurchaseOrder",  
Namespace = "http://examples.microsoft.com/WCF/2005/10/PurchaseOrder")]  
public class PurchaseOrderV1 : IPurchaseOrderV1  
{  
    [DataMember(...)]  
    public string OrderId {...}  
    [DataMember(...)]  
    public string CustomerId {...}  
}  
```  
  
 <span data-ttu-id="00727-238">Hizmet sözleşmesinin işlemleri açısından `PurchaseOrderV1`yazılırken, asıl iş mantığı açısından `IPurchaseOrderV1`olacaktır.</span><span class="sxs-lookup"><span data-stu-id="00727-238">While the service contract’s operations would be written in terms of `PurchaseOrderV1`, the actual business logic would be in terms of `IPurchaseOrderV1`.</span></span> <span data-ttu-id="00727-239">Daha sonra, sürüm 2'de, `IPurchaseOrderV2` aşağıdaki kodda gösterildiği gibi yeni bir arabirim ve yeni `PurchaseOrderV2` bir sınıf olacaktır:</span><span class="sxs-lookup"><span data-stu-id="00727-239">Then, in version 2, there would be a new `IPurchaseOrderV2` interface and a new `PurchaseOrderV2` class as shown in the following code:</span></span>  
  
```csharp
public interface IPurchaseOrderV2  
{  
    DateTime OrderDate { get; set; }  
}

[DataContract(
Name = "PurchaseOrder",  
Namespace = "http://examples.microsoft.com/WCF/2006/02/PurchaseOrder")]  
public class PurchaseOrderV2 : IPurchaseOrderV1, IPurchaseOrderV2  
{  
    [DataMember(...)]  
    public string OrderId {...}  
    [DataMember(...)]  
    public string CustomerId {...}  
    [DataMember(...)]  
    public DateTime OrderDate { ... }  
}  
```  
  
 <span data-ttu-id="00727-240">Hizmet sözleşmesi, `PurchaseOrderV2`'' açısından yazılmış yeni işlemleri içerecek şekilde güncellenecektir.</span><span class="sxs-lookup"><span data-stu-id="00727-240">The service contract would be updated to include new operations that are written in terms of `PurchaseOrderV2`.</span></span> <span data-ttu-id="00727-241">Mevcut iş mantığı açısından `IPurchaseOrderV1` yazılan lar `PurchaseOrderV2` için çalışmaya devam edecek `OrderDate` ve yeni iş `IPurchaseOrderV2`mantığı na ihtiyaç duyduğu özellik açısından yazılacak.</span><span class="sxs-lookup"><span data-stu-id="00727-241">Existing business logic written in terms of `IPurchaseOrderV1` would continue to work for `PurchaseOrderV2` and new business logic that needs the `OrderDate` property would be written in terms of `IPurchaseOrderV2`.</span></span>
