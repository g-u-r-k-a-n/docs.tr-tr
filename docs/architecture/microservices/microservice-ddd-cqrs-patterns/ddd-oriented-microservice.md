### <a name="the-domain-model-layer"></a><span data-ttu-id="bb7a0-159">Etki alanı modeli katmanı</span><span class="sxs-lookup"><span data-stu-id="bb7a0-159">The domain model layer</span></span>

<span data-ttu-id="bb7a0-160">Eric Evans mükemmel kitap [Domain Driven Tasarım](https://domainlanguage.com/ddd/) etki alanı modeli katmanı ve uygulama katmanı hakkında aşağıdaki diyor.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-160">Eric Evans's excellent book [Domain Driven Design](https://domainlanguage.com/ddd/) says the following about the domain model layer and the application layer.</span></span>

<span data-ttu-id="bb7a0-161">**Etki Alanı Modeli Katmanı**: İşletmenin kavramlarını, iş durumu hakkında bilgileri ve iş kurallarını temsil eden sorumludur.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-161">**Domain Model Layer**: Responsible for representing concepts of the business, information about the business situation, and business rules.</span></span> <span data-ttu-id="bb7a0-162">İş durumunu yansıtan devlet, depolamanın teknik detayları altyapıya devredilse de burada kontrol edilir ve kullanılır.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-162">State that reflects the business situation is controlled and used here, even though the technical details of storing it are delegated to the infrastructure.</span></span> <span data-ttu-id="bb7a0-163">Bu katman iş yazılımlarının kalbidir.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-163">This layer is the heart of business software.</span></span>

<span data-ttu-id="bb7a0-164">Etki alanı modeli katmanı, işletmenin ifade edildiği yerdir.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-164">The domain model layer is where the business is expressed.</span></span> <span data-ttu-id="bb7a0-165">.NET'te bir microservice etki alanı modeli katmanı uyguladığınızda, bu katman, veri artı davranışı (mantık lı yöntemler) yakalayan etki alanı varlıklarıyla sınıf kitaplığı olarak kodlanır.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-165">When you implement a microservice domain model layer in .NET, that layer is coded as a class library with the domain entities that capture data plus behavior (methods with logic).</span></span>

<span data-ttu-id="bb7a0-166">[Kalıcılık Cehalet](https://deviq.com/persistence-ignorance/) ve [Altyapı Cehalet](https://ayende.com/blog/3137/infrastructure-ignorance) ilkelerini izleyerek, bu katman tamamen veri kalıcılık ayrıntıları göz ardı etmelidir.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-166">Following the [Persistence Ignorance](https://deviq.com/persistence-ignorance/) and the [Infrastructure Ignorance](https://ayende.com/blog/3137/infrastructure-ignorance) principles, this layer must completely ignore data persistence details.</span></span> <span data-ttu-id="bb7a0-167">Bu kalıcılık görevleri altyapı katmanı tarafından gerçekleştirilmelidir.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-167">These persistence tasks should be performed by the infrastructure layer.</span></span> <span data-ttu-id="bb7a0-168">Bu nedenle, bu katman altyapıya doğrudan bağımlılık lar almamalıdır, bu da önemli bir kuralın etki alanı modeli varlık sınıflarınızın [POCO](https://en.wikipedia.org/wiki/Plain_Old_CLR_Object)s olması gerektiği anlamına gelir.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-168">Therefore, this layer should not take direct dependencies on the infrastructure, which means that an important rule is that your domain model entity classes should be [POCO](https://en.wikipedia.org/wiki/Plain_Old_CLR_Object)s.</span></span>

<span data-ttu-id="bb7a0-169">Etki alanı varlıkları, Entity Framework veya NHibernate gibi herhangi bir veri erişim altyapısı çerçevesinde doğrudan bağımlılığa (taban sınıftan türemiş gibi) sahip olmamalıdır.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-169">Domain entities should not have any direct dependency (like deriving from a base class) on any data access infrastructure framework like Entity Framework or NHibernate.</span></span> <span data-ttu-id="bb7a0-170">İdeal olarak, etki alanı varlıklarınız herhangi bir altyapı çerçevesinde tanımlanan herhangi bir türden türememeli veya uygulamamalıdır.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-170">Ideally, your domain entities should not derive from or implement any type defined in any infrastructure framework.</span></span>

<span data-ttu-id="bb7a0-171">Entity Framework Core gibi çoğu modern ORM çerçevesi bu yaklaşıma izin verir, böylece etki alanı modeli sınıflarınızın altyapıyla birleştiğinde değildir.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-171">Most modern ORM frameworks like Entity Framework Core allow this approach, so that your domain model classes are not coupled to the infrastructure.</span></span> <span data-ttu-id="bb7a0-172">Ancak, Azure Hizmet Dokusunda Aktörler ve Güvenilir Koleksiyonlar gibi belirli NoSQL veritabanlarını ve çerçevelerini kullanırken POCO varlıklarına sahip olmak her zaman mümkün değildir.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-172">However, having POCO entities is not always possible when using certain NoSQL databases and frameworks, like Actors and Reliable Collections in Azure Service Fabric.</span></span>

<span data-ttu-id="bb7a0-173">Etki Alanı modeli niz için Kalıcılık Cehalet ilkesini izlemek önemli olsa bile, kalıcılık endişelerini göz ardı etmemelisiniz.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-173">Even when it is important to follow the Persistence Ignorance principle for your Domain model, you should not ignore persistence concerns.</span></span> <span data-ttu-id="bb7a0-174">Fiziksel veri modelini ve varlık nesnesi modeliyle nasıl eşleşiş olduğunu anlamak yine de çok önemlidir.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-174">It is still very important to understand the physical data model and how it maps to your entity object model.</span></span> <span data-ttu-id="bb7a0-175">Aksi takdirde imkansız tasarımlar oluşturabilirsiniz.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-175">Otherwise you can create impossible designs.</span></span>

<span data-ttu-id="bb7a0-176">Ayrıca, bu ilişkisel bir veritabanı için tasarlanmış bir model alabilir ve doğrudan bir NoSQL veya belge yönelimli veritabanına taşıyabilirsiniz anlamına gelmez.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-176">Also, this does not mean you can take a model designed for a relational database and directly move it to a NoSQL or document-oriented database.</span></span> <span data-ttu-id="bb7a0-177">Bazı varlık modellerinde, model sığabilir, ancak genellikle sığmaz.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-177">In some entity models, the model might fit, but usually it does not.</span></span> <span data-ttu-id="bb7a0-178">Hem depolama teknolojisine hem de ORM teknolojisine dayalı olarak varlık modelinizin uyması gereken kısıtlamalar hala vardır.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-178">There are still constraints that your entity model must adhere to, based both on the storage technology and ORM technology.</span></span>

### <a name="the-application-layer"></a><span data-ttu-id="bb7a0-179">Uygulama katmanı</span><span class="sxs-lookup"><span data-stu-id="bb7a0-179">The application layer</span></span>

<span data-ttu-id="bb7a0-180">Uygulama katmanına geçmeden, yine Eric Evans'ın Domain [Driven Design](https://domainlanguage.com/ddd/)adlı kitabından bahsedebiliriz:</span><span class="sxs-lookup"><span data-stu-id="bb7a0-180">Moving on to the application layer, we can again cite Eric Evans's book [Domain Driven Design](https://domainlanguage.com/ddd/):</span></span>

<span data-ttu-id="bb7a0-181">**Uygulama Katmanı:** Yazılımın yapması gereken işleri tanımlar ve etkileyici etki alanı nesnelerini sorunları çözmesi için yönlendirir.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-181">**Application Layer:** Defines the jobs the software is supposed to do and directs the expressive domain objects to work out problems.</span></span> <span data-ttu-id="bb7a0-182">Bu katmanın sorumlu olduğu görevler işletme için anlamlıdır veya diğer sistemlerin uygulama katmanlarıyla etkileşim için gereklidir.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-182">The tasks this layer is responsible for are meaningful to the business or necessary for interaction with the application layers of other systems.</span></span> <span data-ttu-id="bb7a0-183">Bu tabaka ince tutulur.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-183">This layer is kept thin.</span></span> <span data-ttu-id="bb7a0-184">İş kuralları veya bilgi içermez, ancak yalnızca görevleri ve temsilcileri bir sonraki katmanda etki alanı nesnelerinin işbirlikleri için çalışır koordine eder.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-184">It does not contain business rules or knowledge, but only coordinates tasks and delegates work to collaborations of domain objects in the next layer down.</span></span> <span data-ttu-id="bb7a0-185">İş durumunu yansıtan bir durum yoktur, ancak kullanıcı veya program için bir görevin ilerlemesini yansıtan bir durum olabilir.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-185">It does not have state reflecting the business situation, but it can have state that reflects the progress of a task for the user or the program.</span></span>

<span data-ttu-id="bb7a0-186">Bir microservice'in .NET'teki uygulama katmanı genellikle ASP.NET Core Web API projesi olarak kodlanır.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-186">A microservice's application layer in .NET is commonly coded as an ASP.NET Core Web API project.</span></span> <span data-ttu-id="bb7a0-187">Proje, mikro hizmetin etkileşimini, uzak ağ erişimini ve ui veya istemci uygulamalarından kullanılan harici Web API'lerini uygular.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-187">The project implements the microservice's interaction, remote network access, and the external Web APIs used from the UI or client apps.</span></span> <span data-ttu-id="bb7a0-188">CQRS yaklaşımı kullanıyorsanız sorguları, microservice tarafından kabul edilen komutları ve hatta mikro hizmetler (tümleştirme olayları) arasındaki olay odaklı iletişimi içerir.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-188">It includes queries if using a CQRS approach, commands accepted by the microservice, and even the event-driven communication between microservices (integration events).</span></span> <span data-ttu-id="bb7a0-189">Uygulama katmanını temsil eden ASP.NET Core Web API'si iş kuralları veya etki alanı bilgisi içermemelidir (özellikle hareketler veya güncelleştirmeler için etki alanı kuralları); bunlar etki alanı modeli sınıf kitaplığı tarafından sahip olunmalıdır.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-189">The ASP.NET Core Web API that represents the application layer must not contain business rules or domain knowledge (especially domain rules for transactions or updates); these should be owned by the domain model class library.</span></span> <span data-ttu-id="bb7a0-190">Uygulama katmanı yalnızca görevleri koordine etmeli ve herhangi bir etki alanı durumunu (etki alanı modeli) tutmamalı veya tanımlamamalıdır.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-190">The application layer must only coordinate tasks and must not hold or define any domain state (domain model).</span></span> <span data-ttu-id="bb7a0-191">İş kurallarının yürütülmesini etki alanı modeli sınıflarına (toplu kökler ve etki alanı varlıkları) devreder ve sonuçta bu etki alanı varlıkları içindeki verileri günceller.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-191">It delegates the execution of business rules to the domain model classes themselves (aggregate roots and domain entities), which will ultimately update the data within those domain entities.</span></span>

<span data-ttu-id="bb7a0-192">Temel olarak, uygulama mantığı, belirli bir ön uça bağlı olan tüm kullanım durumlarını uyguladığınız yerdir.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-192">Basically, the application logic is where you implement all use cases that depend on a given front end.</span></span> <span data-ttu-id="bb7a0-193">Örneğin, bir Web API hizmetiyle ilgili uygulama.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-193">For example, the implementation related to a Web API service.</span></span>

<span data-ttu-id="bb7a0-194">Amaç, etki alanı modeli katmanındaki etki alanı mantığının, değişmezlerinin, veri modelinin ve ilgili iş kurallarının sunu ve uygulama katmanlarından tamamen bağımsız olması gerektiğidir.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-194">The goal is that the domain logic in the domain model layer, its invariants, the data model, and related business rules must be completely independent from the presentation and application layers.</span></span> <span data-ttu-id="bb7a0-195">Hepsinden önemlisi, etki alanı modeli katmanı doğrudan herhangi bir altyapı çerçevesine bağlı olmamalıdır.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-195">Most of all, the domain model layer must not directly depend on any infrastructure framework.</span></span>

### <a name="the-infrastructure-layer"></a><span data-ttu-id="bb7a0-196">Altyapı katmanı</span><span class="sxs-lookup"><span data-stu-id="bb7a0-196">The infrastructure layer</span></span>

<span data-ttu-id="bb7a0-197">Altyapı katmanı, başlangıçta etki alanı varlıklarında (bellekte) tutulan verilerin veritabanlarında veya başka bir kalıcı depoda nasıl kalıcı olarak kalıcı olarak kalıcı olarak kalıcı olarak kalıcı olarak bekletildiğidir.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-197">The infrastructure layer is how the data that is initially held in domain entities (in memory) is persisted in databases or another persistent store.</span></span> <span data-ttu-id="bb7a0-198">Bir örnek, İlişkisel bir veritabanındaki verileri kalıcı olarak sürdürmek için DBContext kullanan Depo desen sınıflarını uygulamak için Varlık Framework Core kodunu kullanmaktır.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-198">An example is using Entity Framework Core code to implement the Repository pattern classes that use a DBContext to persist data in a relational database.</span></span>

<span data-ttu-id="bb7a0-199">Daha önce bahsedilen [Kalıcılık Cehalet](https://deviq.com/persistence-ignorance/) ve [Altyapı Cehalet](https://ayende.com/blog/3137/infrastructure-ignorance) ilkeleri uyarınca, altyapı katmanı etki alanı modeli katmanı "kontamine" olmamalıdır.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-199">In accordance with the previously mentioned [Persistence Ignorance](https://deviq.com/persistence-ignorance/) and [Infrastructure Ignorance](https://ayende.com/blog/3137/infrastructure-ignorance) principles, the infrastructure layer must not "contaminate" the domain model layer.</span></span> <span data-ttu-id="bb7a0-200">Çerçevelere sıkı bağımlılıklar yapmayarak, verileri (EF veya başka bir çerçeve) kalıcı tutmak için kullandığınız altyapıdan etki alanı modeli varlık sınıflarını agnostik tutmanız gerekir.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-200">You must keep the domain model entity classes agnostic from the infrastructure that you use to persist data (EF or any other framework) by not taking hard dependencies on frameworks.</span></span> <span data-ttu-id="bb7a0-201">Etki alanı modeli katmanı sınıf kitaplığınız yalnızca etki alanı kodunuza sahip olmalıdır, yalnızca yazılımınızın kalbini uygulayan ve altyapı teknolojilerinden tamamen ayrılmış [OLAN POCO](https://en.wikipedia.org/wiki/Plain_Old_CLR_Object) varlık sınıfları olmalıdır.</span><span class="sxs-lookup"><span data-stu-id="bb7a0-201">Your domain model layer class library should have only your domain code, just [POCO](https://en.wikipedia.org/wiki/Plain_Old_CLR_Object) entity classes implementing the heart of your software and completely decoupled from infrastructure technologies.</span></span>
