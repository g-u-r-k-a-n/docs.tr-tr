# <a name="what-is-managed-code"></a><span data-ttu-id="f1123-103">“Yönetilen kod” nedir?</span><span class="sxs-lookup"><span data-stu-id="f1123-103">What is "managed code"?</span></span>

<span data-ttu-id="f1123-104">.NET Framework ile çalışırken, genellikle "yönetilen kod" terimiyle karşılaşırsınız.</span><span class="sxs-lookup"><span data-stu-id="f1123-104">When working with .NET Framework, you will often encounter the term "managed code".</span></span> <span data-ttu-id="f1123-105">Bu belgede, bu terimin ne anlama geldiğini ve bunun etrafında ek bilgiler açıklanmaktadır.</span><span class="sxs-lookup"><span data-stu-id="f1123-105">This document will explain what this term means and additional information around it.</span></span>

<span data-ttu-id="f1123-106">Çok basit bir şekilde koymak için, yönetilen kod yalnızca bir çalışma zamanı tarafından yönetilen kod olur.</span><span class="sxs-lookup"><span data-stu-id="f1123-106">To put it very simply, managed code is just that: code whose execution is managed by a runtime.</span></span> <span data-ttu-id="f1123-107">Bu durumda, söz konusu çalışma zamanına uygulamadan bağımsız olarak **ortak dil çalışma zamanı** veya clr adı verilir ([mono](https://www.mono-project.com/) veya .NET Framework veya .NET Core).</span><span class="sxs-lookup"><span data-stu-id="f1123-107">In this case, the runtime in question is called the **Common Language Runtime** or CLR, regardless of the implementation ([Mono](https://www.mono-project.com/) or .NET Framework or .NET Core).</span></span> <span data-ttu-id="f1123-108">CLR, yönetilen kodu alma, makine kodunda derleme ve daha sonra yürütme aşamasında olduğundan ücretsizdir.</span><span class="sxs-lookup"><span data-stu-id="f1123-108">CLR is in charge of taking the managed code, compiling it into machine code and then executing it.</span></span> <span data-ttu-id="f1123-109">Bunun üzerine, çalışma zamanı otomatik bellek yönetimi, güvenlik sınırları, tür güvenliği vb. gibi çeşitli önemli hizmetler sağlar.</span><span class="sxs-lookup"><span data-stu-id="f1123-109">On top of that, runtime provides several important services such as automatic memory management, security boundaries, type safety etc.</span></span>

<span data-ttu-id="f1123-110">Bunu, "yönetilmeyen kod" olarak da adlandırılan bir C/C++ programını çalıştırdığınız şekilde kontrast.</span><span class="sxs-lookup"><span data-stu-id="f1123-110">Contrast this to the way you would run a C/C++ program, also called "unmanaged code".</span></span> <span data-ttu-id="f1123-111">Yönetilmeyen dünyada, programcı çok sayıda her şeyin üzerinden ücretlendirilir.</span><span class="sxs-lookup"><span data-stu-id="f1123-111">In the unmanaged world, the programmer is in charge of pretty much everything.</span></span> <span data-ttu-id="f1123-112">Gerçek program temelde, işletim sisteminin (OS) belleğe yüklediği ve başladığı bir ikili dosya olur.</span><span class="sxs-lookup"><span data-stu-id="f1123-112">The actual program is, essentially, a binary that the operating system (OS) loads into memory and starts.</span></span> <span data-ttu-id="f1123-113">Bellek yönetiminden güvenlik 'e kadar olan diğer her şey programcı 'nin bir yükünden oluşur.</span><span class="sxs-lookup"><span data-stu-id="f1123-113">Everything else, from memory management to security considerations are a burden of the programmer.</span></span>

<span data-ttu-id="f1123-114">Yönetilen kod, C#, Visual Basic, F # ve diğerleri gibi .NET üzerinde çalıştırılabilen üst düzey dillerden birinde yazılır.</span><span class="sxs-lookup"><span data-stu-id="f1123-114">Managed code is written in one of the high-level languages that can be run on top of .NET, such as C#, Visual Basic, F# and others.</span></span> <span data-ttu-id="f1123-115">Bu dillerde yazılmış kodu kendi derleyicisinden derlerken, makine kodu almaz.</span><span class="sxs-lookup"><span data-stu-id="f1123-115">When you compile code written in those languages with their respective compiler, you don’t get machine code.</span></span> <span data-ttu-id="f1123-116">Çalışma zamanının daha sonra derleyen ve yürütülen **ara dil** kodunu alırsınız.</span><span class="sxs-lookup"><span data-stu-id="f1123-116">You get **Intermediate Language** code which the runtime then compiles and executes.</span></span> <span data-ttu-id="f1123-117">C++, Windows üzerinde çalışan yerel, yönetilmeyen ikili dosyalar da oluşturabildiği için bu kuralın tek istisnası olur.</span><span class="sxs-lookup"><span data-stu-id="f1123-117">C++ is the one exception to this rule, as it can also produce native, unmanaged binaries that run on Windows.</span></span>

## <a name="intermediate-language--execution"></a><span data-ttu-id="f1123-118">Ara dil & yürütme</span><span class="sxs-lookup"><span data-stu-id="f1123-118">Intermediate Language & execution</span></span>

<span data-ttu-id="f1123-119">"Ara dil" (veya kısa için Il) nedir?</span><span class="sxs-lookup"><span data-stu-id="f1123-119">What is "Intermediate Language" (or IL for short)?</span></span> <span data-ttu-id="f1123-120">Bu, üst düzey .NET dillerde yazılmış kodların derlenmesi ürünüdür.</span><span class="sxs-lookup"><span data-stu-id="f1123-120">It is a product of compilation of code written in high-level .NET languages.</span></span> <span data-ttu-id="f1123-121">Bu dillerden birinde yazılan kodunuzu derledikten sonra Il 'den çıkan bir ikiliye sahip olursunuz.</span><span class="sxs-lookup"><span data-stu-id="f1123-121">Once you compile your code written in one of these languages, you will get a binary that is made out of IL.</span></span> <span data-ttu-id="f1123-122">Il 'nin çalışma zamanının üzerinde çalışan herhangi bir dilden bağımsız olduğunu unutmamak önemlidir; Bu durumda, daha fazla saysanız okuyabilmeniz için ayrı bir belirtim de vardır.</span><span class="sxs-lookup"><span data-stu-id="f1123-122">It is important to note that the IL is independent from any specific language that runs on top of the runtime; there is even a separate specification for it that you can read if you’re so inclined.</span></span>

<span data-ttu-id="f1123-123">Üst düzey kodınızdan Il 'yi oluşturduktan sonra büyük olasılıkla çalıştırmak istersiniz.</span><span class="sxs-lookup"><span data-stu-id="f1123-123">Once you produce IL from your high-level code, you will most likely want to run it.</span></span> <span data-ttu-id="f1123-124">Bu, CLR 'nin **tam zamanında** derleme işlemini nereden aldığını ve başlattığı veya bir CPU üzerinde gerçekten ÇALıŞTıRıLABILECEK şekilde Il 'den makine koduna kodlalama işleminin bulunduğu yerdir. **JIT-ing**</span><span class="sxs-lookup"><span data-stu-id="f1123-124">This is where the CLR takes over and starts the process of **Just-In-Time** compiling, or **JIT-ing** your code from IL to machine code that can actually be run on a CPU.</span></span> <span data-ttu-id="f1123-125">Bu şekilde, CLR kodunuzun ne yaptığını tam olarak bilir ve onu etkili bir şekilde _yönetebilir_ .</span><span class="sxs-lookup"><span data-stu-id="f1123-125">In this way, the CLR knows exactly what your code is doing and can effectively _manage_ it.</span></span>

<span data-ttu-id="f1123-126">Ara dil bazen ortak ara dil (CıL) veya Microsoft ara dili (MSIL) olarak da adlandırılır.</span><span class="sxs-lookup"><span data-stu-id="f1123-126">Intermediate Language is sometimes also called Common Intermediate Language (CIL) or Microsoft Intermediate Language (MSIL).</span></span>

## <a name="unmanaged-code-interoperability"></a><span data-ttu-id="f1123-127">Yönetilmeyen kod birlikte çalışabilirliği</span><span class="sxs-lookup"><span data-stu-id="f1123-127">Unmanaged code interoperability</span></span>

<span data-ttu-id="f1123-128">Kuşkusuz, CLR yönetilen ve yönetilmeyen dünya arasındaki sınırları geçirmeyi sağlar ve [temel sınıf kitaplıklarında](framework-libraries.md)bile bu şekilde çok sayıda kod vardır.</span><span class="sxs-lookup"><span data-stu-id="f1123-128">Of course, the CLR allows passing the boundaries between managed and unmanaged world, and there is a lot of code that does that, even in the [Base Class Libraries](framework-libraries.md).</span></span> <span data-ttu-id="f1123-129">Bu, **birlikte çalışabilirlik** veya kısa bir şekilde **birlikte** çalışabilirlik olarak adlandırılır.</span><span class="sxs-lookup"><span data-stu-id="f1123-129">This is called **interoperability** or just **interop** for short.</span></span> <span data-ttu-id="f1123-130">Bu hükümler, örneğin, yönetilmeyen bir kitaplığı kaydırabilmeniz ve buna çağrı yapmanıza olanak sağlar.</span><span class="sxs-lookup"><span data-stu-id="f1123-130">These provisions would allow you to, for example, wrap up an unmanaged library and call into it.</span></span> <span data-ttu-id="f1123-131">Ancak, bunu yaptığınızda, kod çalışma zamanının sınırlarını geçtiğinde, yürütmenin gerçek yönetiminin, yönetilmeyen kodun yanında yeniden olduğunu ve bu nedenle aynı kısıtlamaların altına düştüğünü unutmayın.</span><span class="sxs-lookup"><span data-stu-id="f1123-131">However, it is important to note that once you do this, when the code passes the boundaries of the runtime, the actual management of the execution is again in the hand of unmanaged code, and thus falls under the same restrictions.</span></span>

<span data-ttu-id="f1123-132">Buna benzer şekilde C#, yürütmenin CLR tarafından yönetilmediği bir kod parçasını atayan, **güvenli olmayan bağlam** olarak bilinerek doğrudan kodda işaretçiler gibi yönetilmeyen yapıları kullanmanıza olanak tanıyan bir dildir.</span><span class="sxs-lookup"><span data-stu-id="f1123-132">Similar to this, C# is one language that allows you to use unmanaged constructs such as pointers directly in code by utilizing what is known as **unsafe context** which designates a piece of code for which the execution is not managed by the CLR.</span></span>
