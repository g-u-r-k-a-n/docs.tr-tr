# <a name="extension-methods"></a><span data-ttu-id="f4135-102">Uzantı Metotları</span><span class="sxs-lookup"><span data-stu-id="f4135-102">Extension Methods</span></span>
<span data-ttu-id="f4135-103">Uzantı yöntemleri, statik yöntemlerin örnek yöntemi çağırma sözdizimi kullanılarak çağrılmasına izin veren bir dil özelliğidir.</span><span class="sxs-lookup"><span data-stu-id="f4135-103">Extension methods are a language feature that allows static methods to be called using instance method call syntax.</span></span> <span data-ttu-id="f4135-104">Bu yöntemler, yöntemin üzerinde çalışacağı örneği temsil eden en az bir parametre almalıdır.</span><span class="sxs-lookup"><span data-stu-id="f4135-104">These methods must take at least one parameter, which represents the instance the method is to operate on.</span></span>

 <span data-ttu-id="f4135-105">Bu tür uzantı yöntemlerini tanımlayan sınıf "sponsor" sınıfı olarak adlandırılır ve statik olarak bildirilmelidir.</span><span class="sxs-lookup"><span data-stu-id="f4135-105">The class that defines such extension methods is referred to as the "sponsor" class, and it must be declared as static.</span></span> <span data-ttu-id="f4135-106">Uzantı yöntemlerini kullanmak için, bir tane sponsor sınıfı tanımlayan ad alanını içeri aktarmanız gerekir.</span><span class="sxs-lookup"><span data-stu-id="f4135-106">To use extension methods, one must import the namespace defining the sponsor class.</span></span>

 <span data-ttu-id="f4135-107">❌ Özellikle sahip olmadığınız türler üzerinde uzantı yöntemlerini tanımlamayı frivolously KAÇıNıN.</span><span class="sxs-lookup"><span data-stu-id="f4135-107">❌ AVOID frivolously defining extension methods, especially on types you don't own.</span></span>

 <span data-ttu-id="f4135-108">Bir türün kaynak kodunu yaptıysanız bunun yerine normal örnek yöntemleri kullanmayı düşünün.</span><span class="sxs-lookup"><span data-stu-id="f4135-108">If you do own source code of a type, consider using regular instance methods instead.</span></span> <span data-ttu-id="f4135-109">Sahip değilseniz ve bir yöntem eklemek istiyorsanız, çok dikkatli olun.</span><span class="sxs-lookup"><span data-stu-id="f4135-109">If you don't own, and you want to add a method, be very careful.</span></span> <span data-ttu-id="f4135-110">Uzantı yöntemlerinin serbest kullanımı, bu yöntemlere sahip olmak için tasarlanmamış türlerin karışıklık API 'Lerinin potansiyelini içerir.</span><span class="sxs-lookup"><span data-stu-id="f4135-110">Liberal use of extension methods has the potential of cluttering APIs of types that were not designed to have these methods.</span></span>

 <span data-ttu-id="f4135-111">✔️ Aşağıdaki senaryolardan birinde uzantı yöntemlerini kullanmayı göz önünde bulundurun:</span><span class="sxs-lookup"><span data-stu-id="f4135-111">✔️ CONSIDER using extension methods in any of the following scenarios:</span></span>

- <span data-ttu-id="f4135-112">Bir arabirimin her uygulamasıyla ilgili yardımcı işlevsellik sağlamak için, Eğer işlevin çekirdek arabirim açısından yazılabilmesini sağlar.</span><span class="sxs-lookup"><span data-stu-id="f4135-112">To provide helper functionality relevant to every implementation of an interface, if said functionality can be written in terms of the core interface.</span></span> <span data-ttu-id="f4135-113">Bunun nedeni, somut uygulamaların arabirime atanamadığı bir şekilde yapılamaz.</span><span class="sxs-lookup"><span data-stu-id="f4135-113">This is because concrete implementations cannot otherwise be assigned to interfaces.</span></span> <span data-ttu-id="f4135-114">Örneğin, `LINQ to Objects` işleçler tüm türler için uzantı yöntemleri olarak uygulanır <xref:System.Collections.Generic.IEnumerable%601> .</span><span class="sxs-lookup"><span data-stu-id="f4135-114">For example, the `LINQ to Objects` operators are implemented as extension methods for all <xref:System.Collections.Generic.IEnumerable%601> types.</span></span> <span data-ttu-id="f4135-115">Bu nedenle, herhangi bir `IEnumerable<>` uygulama otomatik olarak LINQ etkindir.</span><span class="sxs-lookup"><span data-stu-id="f4135-115">Thus, any `IEnumerable<>` implementation is automatically LINQ-enabled.</span></span>

- <span data-ttu-id="f4135-116">Bir örnek yöntemi, bazı tür için bir bağımlılık tanıtacaksa, ancak böyle bir bağımlılık, bağımlılık yönetimi kurallarını kesintiye uğratır.</span><span class="sxs-lookup"><span data-stu-id="f4135-116">When an instance method would introduce a dependency on some type, but such a dependency would break dependency management rules.</span></span> <span data-ttu-id="f4135-117">Örneğin, öğesinden bir bağımlılığı <xref:System.String> <xref:System.Uri?displayProperty=nameWithType> muhtemelen istenmez ve bu nedenle `String.ToUri()` döndürülen örnek yöntemi `System.Uri` bir bağımlılık yönetimi perspektifinden yanlış tasarım olacaktır.</span><span class="sxs-lookup"><span data-stu-id="f4135-117">For example, a dependency from <xref:System.String> to <xref:System.Uri?displayProperty=nameWithType> is probably not desirable, and so `String.ToUri()` instance method returning `System.Uri` would be the wrong design from a dependency management perspective.</span></span> <span data-ttu-id="f4135-118">Döndüren statik bir genişletme `Uri.ToUri(this string str)` yöntemi `System.Uri` çok daha iyi bir tasarım olacaktır.</span><span class="sxs-lookup"><span data-stu-id="f4135-118">A static extension method `Uri.ToUri(this string str)` returning `System.Uri` would be a much better design.</span></span>

 <span data-ttu-id="f4135-119">❌ Uzantı yöntemlerini tanımlamaktan KAÇıNıN <xref:System.Object?displayProperty=nameWithType> .</span><span class="sxs-lookup"><span data-stu-id="f4135-119">❌ AVOID defining extension methods on <xref:System.Object?displayProperty=nameWithType>.</span></span>

 <span data-ttu-id="f4135-120">VB kullanıcıları, uzantı yöntemi sözdizimini kullanarak nesne başvuruları üzerinde bu tür yöntemleri çağıramayacak.</span><span class="sxs-lookup"><span data-stu-id="f4135-120">VB users will not be able to call such methods on object references using the extension method syntax.</span></span> <span data-ttu-id="f4135-121">VB, nesne olarak bir başvuru bildirmek, bu tür yöntemlerin çağrılmasını desteklemez, çünkü VB 'de, nesne üzerinde tüm yöntem etkinleştirmeleri, geç bağlama olur (çağrılan gerçek üye çalışma zamanında belirlenir), genişletme yöntemlerine bağlamalar derleme zamanı (erken bağlantı) sırasında belirlenir.</span><span class="sxs-lookup"><span data-stu-id="f4135-121">VB does not support calling such methods because, in VB, declaring a reference as Object forces all method invocations on it to be late bound (actual member called is determined at runtime), while bindings to extension methods are determined at compile-time (early bound).</span></span>

 <span data-ttu-id="f4135-122">Kılavuz, aynı bağlama davranışının bulunduğu ya da uzantı yöntemlerinin desteklenmediği diğer dillere uygulanacağını unutmayın.</span><span class="sxs-lookup"><span data-stu-id="f4135-122">Note that the guideline applies to other languages where the same binding behavior is present, or where extension methods are not supported.</span></span>

 <span data-ttu-id="f4135-123">❌ Arayüzlere veya bağımlılık yönetimine Yöntemler eklenmediği sürece uzantı yöntemlerini genişletilmiş türle aynı ad alanına yerleştirmeyin.</span><span class="sxs-lookup"><span data-stu-id="f4135-123">❌ DO NOT put extension methods in the same namespace as the extended type unless it is for adding methods to interfaces or for dependency management.</span></span>

 <span data-ttu-id="f4135-124">❌ Farklı ad alanlarında bulunsa bile, aynı imzaya sahip iki veya daha fazla genişletme yöntemi tanımlamaktan KAÇıNıN.</span><span class="sxs-lookup"><span data-stu-id="f4135-124">❌ AVOID defining two or more extension methods with the same signature, even if they reside in different namespaces.</span></span>

 <span data-ttu-id="f4135-125">✔️, tür bir arabirim ise ve uzantı yöntemlerinin çoğu durumda veya tüm durumlarda kullanılması amaçlanan uzantı yöntemlerini genişletilmiş türle tanımlamayı düşünün.</span><span class="sxs-lookup"><span data-stu-id="f4135-125">✔️ CONSIDER defining extension methods in the same namespace as the extended type if the type is an interface and if the extension methods are meant to be used in most or all cases.</span></span>

 <span data-ttu-id="f4135-126">❌ Genellikle diğer özelliklerle ilişkili ad alanlarında bir özelliği uygulayan uzantı yöntemlerini tanımlamayın.</span><span class="sxs-lookup"><span data-stu-id="f4135-126">❌ DO NOT define extension methods implementing a feature in namespaces normally associated with other features.</span></span> <span data-ttu-id="f4135-127">Bunun yerine, bunları ait oldukları özellikle ilişkili ad alanı içinde tanımlayın.</span><span class="sxs-lookup"><span data-stu-id="f4135-127">Instead, define them in the namespace associated with the feature they belong to.</span></span>

 <span data-ttu-id="f4135-128">❌ Uzantı yöntemlerine adanmış ad alanlarının genel olarak adlandırılmasını ÖNLEYIN (ör. "Uzantılar").</span><span class="sxs-lookup"><span data-stu-id="f4135-128">❌ AVOID generic naming of namespaces dedicated to extension methods (e.g., "Extensions").</span></span> <span data-ttu-id="f4135-129">Bunun yerine açıklayıcı bir ad (örneğin, "yönlendirme") kullanın.</span><span class="sxs-lookup"><span data-stu-id="f4135-129">Use a descriptive name (e.g., "Routing") instead.</span></span>
