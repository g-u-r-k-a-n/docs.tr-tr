# <a name="what-is-managed-code"></a><span data-ttu-id="80b5a-103">“Yönetilen kod” nedir?</span><span class="sxs-lookup"><span data-stu-id="80b5a-103">What is "managed code"?</span></span>

<span data-ttu-id="80b5a-104">.NET Framework ile çalışırken sık sık "yönetilen kod" terimiyle karşılaşırsınız.</span><span class="sxs-lookup"><span data-stu-id="80b5a-104">When working with .NET Framework, you will often encounter the term "managed code".</span></span> <span data-ttu-id="80b5a-105">Bu belge, bu terimin ne anlama geldiğini ve çevresinde ek bilgiler açıklayacaktır.</span><span class="sxs-lookup"><span data-stu-id="80b5a-105">This document will explain what this term means and additional information around it.</span></span>

<span data-ttu-id="80b5a-106">Çok basit bir ifadeyle, yönetilen kod sadece şudur: yürütmesi bir çalışma zamanı tarafından yönetilen kod.</span><span class="sxs-lookup"><span data-stu-id="80b5a-106">To put it very simply, managed code is just that: code whose execution is managed by a runtime.</span></span> <span data-ttu-id="80b5a-107">Bu durumda, söz konusu çalışma süresi, uygulamadan bağımsız olarak[(Mono](https://www.mono-project.com/) veya .NET Framework veya .NET Core) **Ortak Dil Çalışma Süresi** veya CLR olarak adlandırılır.</span><span class="sxs-lookup"><span data-stu-id="80b5a-107">In this case, the runtime in question is called the **Common Language Runtime** or CLR, regardless of the implementation ([Mono](https://www.mono-project.com/) or .NET Framework or .NET Core).</span></span> <span data-ttu-id="80b5a-108">CLR, yönetilen kodu almaktan, makine koduna derlemekten ve daha sonra yürütmekten sorumludur.</span><span class="sxs-lookup"><span data-stu-id="80b5a-108">CLR is in charge of taking the managed code, compiling it into machine code and then executing it.</span></span> <span data-ttu-id="80b5a-109">Bunun üzerine, çalışma süresi otomatik bellek yönetimi, güvenlik sınırları, tip güvenliği vb. gibi birçok önemli hizmeti sağlar.</span><span class="sxs-lookup"><span data-stu-id="80b5a-109">On top of that, runtime provides several important services such as automatic memory management, security boundaries, type safety etc.</span></span>

<span data-ttu-id="80b5a-110">Bunu, "yönetilmeyen kod" olarak da adlandırılan bir C/C++ programını çalıştırma şeklinizle karşılaştırın.</span><span class="sxs-lookup"><span data-stu-id="80b5a-110">Contrast this to the way you would run a C/C++ program, also called "unmanaged code".</span></span> <span data-ttu-id="80b5a-111">Yönetilmeyen dünyada, programcı hemen hemen her şeyden sorumludur.</span><span class="sxs-lookup"><span data-stu-id="80b5a-111">In the unmanaged world, the programmer is in charge of pretty much everything.</span></span> <span data-ttu-id="80b5a-112">Gerçek program, aslında, bir ikili işletim sistemi (OS) bellek içine yükler ve başlar.</span><span class="sxs-lookup"><span data-stu-id="80b5a-112">The actual program is, essentially, a binary that the operating system (OS) loads into memory and starts.</span></span> <span data-ttu-id="80b5a-113">Bellek yönetiminden güvenlik konularına kadar her şey programcının yükündedir.</span><span class="sxs-lookup"><span data-stu-id="80b5a-113">Everything else, from memory management to security considerations are a burden of the programmer.</span></span>

<span data-ttu-id="80b5a-114">Yönetilen kod, C#, Visual Basic, F# ve diğerleri gibi .NET'in üstünde çalıştırılabilen üst düzey dillerden birinde yazılır.</span><span class="sxs-lookup"><span data-stu-id="80b5a-114">Managed code is written in one of the high-level languages that can be run on top of .NET, such as C#, Visual Basic, F# and others.</span></span> <span data-ttu-id="80b5a-115">Bu dillerde kendi derleyicileriyle yazılmış kodları derlediğinizde, makine kodu alamazsınız.</span><span class="sxs-lookup"><span data-stu-id="80b5a-115">When you compile code written in those languages with their respective compiler, you don’t get machine code.</span></span> <span data-ttu-id="80b5a-116">Çalışma zamanının derlediği ve çalıştırdığı **Ara Dil** kodunu alırsınız.</span><span class="sxs-lookup"><span data-stu-id="80b5a-116">You get **Intermediate Language** code which the runtime then compiles and executes.</span></span> <span data-ttu-id="80b5a-117">C++ bu kuralın tek istisnasıdır, çünkü Windows'ta çalışan yerel, yönetilmeyen ikili ler de üretebilir.</span><span class="sxs-lookup"><span data-stu-id="80b5a-117">C++ is the one exception to this rule, as it can also produce native, unmanaged binaries that run on Windows.</span></span>

## <a name="intermediate-language--execution"></a><span data-ttu-id="80b5a-118">Orta Dil & yürütme</span><span class="sxs-lookup"><span data-stu-id="80b5a-118">Intermediate Language & execution</span></span>

<span data-ttu-id="80b5a-119">"Orta Dil" (veya kısaca IL) nedir?</span><span class="sxs-lookup"><span data-stu-id="80b5a-119">What is "Intermediate Language" (or IL for short)?</span></span> <span data-ttu-id="80b5a-120">Üst düzey .NET dillerinde yazılmış kodun bir derleme ürünüdür.</span><span class="sxs-lookup"><span data-stu-id="80b5a-120">It is a product of compilation of code written in high-level .NET languages.</span></span> <span data-ttu-id="80b5a-121">Bu dillerden birinde yazılmış kodunuzu derledikten sonra, IL'den yapılmış bir ikili alırsınız.</span><span class="sxs-lookup"><span data-stu-id="80b5a-121">Once you compile your code written in one of these languages, you will get a binary that is made out of IL.</span></span> <span data-ttu-id="80b5a-122">IL'nin çalışma zamanının üstünde çalışan herhangi bir dilden bağımsız olduğunu unutmayın; eğer çok eğimli iseniz okuyabilirsiniz bunun için ayrı bir belirtim bile vardır.</span><span class="sxs-lookup"><span data-stu-id="80b5a-122">It is important to note that the IL is independent from any specific language that runs on top of the runtime; there is even a separate specification for it that you can read if you’re so inclined.</span></span>

<span data-ttu-id="80b5a-123">Üst düzey kodunuzdan IL'yi ürettiğinizde, büyük olasılıkla çalıştırmak isteyeceksiniz.</span><span class="sxs-lookup"><span data-stu-id="80b5a-123">Once you produce IL from your high-level code, you will most likely want to run it.</span></span> <span data-ttu-id="80b5a-124">Burası CLR'nin devreye girip **Just-In-Time** derleme işlemini başlattığı veya kodunuzu IL'den cpu üzerinde çalıştırılabilen makine koduna **jit-ing** işlemini başlattığı yerdir.</span><span class="sxs-lookup"><span data-stu-id="80b5a-124">This is where the CLR takes over and starts the process of **Just-In-Time** compiling, or **JIT-ing** your code from IL to machine code that can actually be run on a CPU.</span></span> <span data-ttu-id="80b5a-125">Bu şekilde, CLR kodunuzun tam olarak ne yaptığını bilir ve etkili bir şekilde _yönetebilir._</span><span class="sxs-lookup"><span data-stu-id="80b5a-125">In this way, the CLR knows exactly what your code is doing and can effectively _manage_ it.</span></span>

<span data-ttu-id="80b5a-126">Ara Dil bazen Ortak Ara Dil (CIL) veya Microsoft Intermediate Language (MSIL) olarak da adlandırılır.</span><span class="sxs-lookup"><span data-stu-id="80b5a-126">Intermediate Language is sometimes also called Common Intermediate Language (CIL) or Microsoft Intermediate Language (MSIL).</span></span>

## <a name="unmanaged-code-interoperability"></a><span data-ttu-id="80b5a-127">Yönetilmeyen kod birlikte çalışabilirliği</span><span class="sxs-lookup"><span data-stu-id="80b5a-127">Unmanaged code interoperability</span></span>

<span data-ttu-id="80b5a-128">Tabii ki, CLR yönetilen ve yönetilmeyen dünya arasındaki sınırları geçen sağlar ve [base class kütüphaneleri](framework-libraries.md)bile, bunu kod bir yeri vardır.</span><span class="sxs-lookup"><span data-stu-id="80b5a-128">Of course, the CLR allows passing the boundaries between managed and unmanaged world, and there is a lot of code that does that, even in the [Base Class Libraries](framework-libraries.md).</span></span> <span data-ttu-id="80b5a-129">Buna **birlikte çalışabilirlik** ya da kısaca **interop** denir.</span><span class="sxs-lookup"><span data-stu-id="80b5a-129">This is called **interoperability** or just **interop** for short.</span></span> <span data-ttu-id="80b5a-130">Bu hükümler, örneğin, yönetilmeyen bir kitaplığı tamamlamanızı ve bu kitapiçin aramanızı sağlar.</span><span class="sxs-lookup"><span data-stu-id="80b5a-130">These provisions would allow you to, for example, wrap up an unmanaged library and call into it.</span></span> <span data-ttu-id="80b5a-131">Ancak, bunu yaptığınızda, kod çalışma zamanının sınırlarını geçtiğinde, yürütmenin gerçek yönetiminin yine yönetilmeyen kodun elinde olduğunu ve böylece aynı kısıtlamalar altında kaldığını unutmayın.</span><span class="sxs-lookup"><span data-stu-id="80b5a-131">However, it is important to note that once you do this, when the code passes the boundaries of the runtime, the actual management of the execution is again in the hand of unmanaged code, and thus falls under the same restrictions.</span></span>

<span data-ttu-id="80b5a-132">Buna benzer şekilde, C# yürütme CLR tarafından yönetilmeyen bir kod parçası belirleyen güvenli olmayan **bağlam** olarak bilinen kullanarak doğrudan kod işaretçileri gibi yönetilmeyen yapıları kullanmanızı sağlayan bir dildir.</span><span class="sxs-lookup"><span data-stu-id="80b5a-132">Similar to this, C# is one language that allows you to use unmanaged constructs such as pointers directly in code by utilizing what is known as **unsafe context** which designates a piece of code for which the execution is not managed by the CLR.</span></span>
