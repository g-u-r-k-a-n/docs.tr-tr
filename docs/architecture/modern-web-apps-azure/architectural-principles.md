### <a name="explicit-dependencies"></a><span data-ttu-id="75cff-142">Açık bağımlılıklar</span><span class="sxs-lookup"><span data-stu-id="75cff-142">Explicit dependencies</span></span>

### <a name="single-responsibility"></a><span data-ttu-id="75cff-149">Tek sorumluluk</span><span class="sxs-lookup"><span data-stu-id="75cff-149">Single responsibility</span></span>

<span data-ttu-id="75cff-166">Mantığı çoğaltmak yerine bir programlama yapısında kapsülleyebilirsiniz.</span><span class="sxs-lookup"><span data-stu-id="75cff-166">Rather than duplicating logic, encapsulate it in a programming construct.</span></span> <span data-ttu-id="75cff-167">Bu davranış üzerinde tek bir yetki oluşturun ve uygulamanın bu davranışı gerektiren başka bir bölümü yeni yapıyı kullanın.</span><span class="sxs-lookup"><span data-stu-id="75cff-167">Make this construct the single authority over this behavior, and have any other part of the application that requires this behavior use the new construct.</span></span>

> <span data-ttu-id="75cff-168">Yalnızca tesadüfen yinelenen bir davranışı bağlamaktan kaçının.</span><span class="sxs-lookup"><span data-stu-id="75cff-168">Avoid binding together behavior that is only coincidentally repetitive.</span></span> <span data-ttu-id="75cff-169">Örneğin, yalnızca iki farklı sabit değerin aynı değere sahip olması nedeniyle, kavramsal olarak farklı şeylere başvurduklarında yalnızca bir sabit değeri olması gerekir.</span><span class="sxs-lookup"><span data-stu-id="75cff-169">For example, just because two different constants both have the same value, that doesn't mean you should have only one constant, if conceptually they're referring to different things.</span></span>

### <a name="persistence-ignorance"></a><span data-ttu-id="75cff-170">Kalıcılık Ignorance</span><span class="sxs-lookup"><span data-stu-id="75cff-170">Persistence ignorance</span></span>

<span data-ttu-id="75cff-171">**Kalıcılık Ignorance** (PI), kalıcı olması gereken türlere başvurur, ancak kodu Kalıcılık teknolojisi seçimi tarafından etkilenmemiştir.</span><span class="sxs-lookup"><span data-stu-id="75cff-171">**Persistence ignorance** (PI) refers to types that need to be persisted, but whose code is unaffected by the choice of persistence technology.</span></span> <span data-ttu-id="75cff-172">.NET 'teki bu tür türler, belirli bir temel sınıftan devralması veya belirli bir arabirim uygulamak zorunda olmadıkları için bazen düz eski CLR nesneleri (POCOs) olarak adlandırılır.</span><span class="sxs-lookup"><span data-stu-id="75cff-172">Such types in .NET are sometimes referred to as Plain Old CLR Objects (POCOs), because they do not need to inherit from a particular base class or implement a particular interface.</span></span> <span data-ttu-id="75cff-173">Aynı iş modelinin birden çok şekilde kalıcı olmasını sağladığından, uygulama için ek esneklik sunarak Kalıcılık kalıcılığı önem taşır.</span><span class="sxs-lookup"><span data-stu-id="75cff-173">Persistence ignorance is valuable because it allows the same business model to be persisted in multiple ways, offering additional flexibility to the application.</span></span> <span data-ttu-id="75cff-174">Kalıcılık seçimleri zaman içinde, bir veritabanı teknolojisinden diğerine değişebilir ya da uygulamanın başladığı her şeyi (örneğin, bir Redsıs önbelleği veya bir ilişkisel veritabanına ek olarak Azure Cosmos DB) ek olarak başka Kalıcılık biçimleri de gerekebilir.</span><span class="sxs-lookup"><span data-stu-id="75cff-174">Persistence choices might change over time, from one database technology to another, or additional forms of persistence might be required in addition to whatever the application started with (for example, using a Redis cache or Azure Cosmos DB in addition to a relational database).</span></span>

<span data-ttu-id="75cff-175">Bu ilkeye yönelik ihlallere örnek olarak şunlar verilebilir:</span><span class="sxs-lookup"><span data-stu-id="75cff-175">Some examples of violations of this principle include:</span></span>

- <span data-ttu-id="75cff-176">Gerekli bir temel sınıf.</span><span class="sxs-lookup"><span data-stu-id="75cff-176">A required base class.</span></span>

- <span data-ttu-id="75cff-177">Gerekli bir arabirim uygulamasıdır.</span><span class="sxs-lookup"><span data-stu-id="75cff-177">A required interface implementation.</span></span>

- <span data-ttu-id="75cff-178">Kendilerini kaydetmekten sorumlu sınıflar (örneğin, etkin kayıt deseninin).</span><span class="sxs-lookup"><span data-stu-id="75cff-178">Classes responsible for saving themselves (such as the Active Record pattern).</span></span>

- <span data-ttu-id="75cff-179">Parametresiz Oluşturucu gerekli.</span><span class="sxs-lookup"><span data-stu-id="75cff-179">Required parameterless constructor.</span></span>

- <span data-ttu-id="75cff-180">Sanal anahtar sözcük gerektiren özellikler.</span><span class="sxs-lookup"><span data-stu-id="75cff-180">Properties requiring virtual keyword.</span></span>

- <span data-ttu-id="75cff-181">Kalıcılığa özgü gerekli öznitelikler.</span><span class="sxs-lookup"><span data-stu-id="75cff-181">Persistence-specific required attributes.</span></span>

<span data-ttu-id="75cff-182">Sınıfların yukarıdaki özelliklerden veya davranışların herhangi birine sahip olması, kalıcı hale getirilmesi gereken türler ve kalıcılık teknolojileri seçimi arasında bir kuponu ekleyerek yeni veri erişim stratejilerini daha da benimsemeyi daha kolay hale getirir.</span><span class="sxs-lookup"><span data-stu-id="75cff-182">The requirement that classes have any of the above features or behaviors adds coupling between the types to be persisted and the choice of persistence technology, making it more difficult to adopt new data access strategies in the future.</span></span>

### <a name="bounded-contexts"></a><span data-ttu-id="75cff-183">Sınırlanmış bağlamlar</span><span class="sxs-lookup"><span data-stu-id="75cff-183">Bounded contexts</span></span>

<span data-ttu-id="75cff-184">**Sınırlanmış bağlamlar** Domain-Driven tasarımında merkezi bir modeldir.</span><span class="sxs-lookup"><span data-stu-id="75cff-184">**Bounded contexts** are a central pattern in Domain-Driven Design.</span></span> <span data-ttu-id="75cff-185">Bu kişiler, büyük uygulamalarda veya kuruluşlarda, farklı kavramsal modüllere kadar bölmek için bir yol sağlar.</span><span class="sxs-lookup"><span data-stu-id="75cff-185">They provide a way of tackling complexity in large applications or organizations by breaking it up into separate conceptual modules.</span></span> <span data-ttu-id="75cff-186">Her kavramsal modül daha sonra diğer bağlamlardan ayrılan (Bu nedenle, sınırlı) bir bağlamı temsil eder ve bağımsız olarak gelişebilirler.</span><span class="sxs-lookup"><span data-stu-id="75cff-186">Each conceptual module then represents a context that is separated from other contexts (hence, bounded), and can evolve independently.</span></span> <span data-ttu-id="75cff-187">Her sınırlanmış bağlam, içindeki kavramlar için kendi adlarını seçmek üzere ideal olmalıdır ve kendi kalıcılık deposuna özel erişime sahip olmalıdır.</span><span class="sxs-lookup"><span data-stu-id="75cff-187">Each bounded context should ideally be free to choose its own names for concepts within it, and should have exclusive access to its own persistence store.</span></span>

<span data-ttu-id="75cff-188">En azından, bireysel Web uygulamaları, bir veritabanını diğer uygulamalarla paylaşmak yerine kendi iş modeliyle ilgili kendi Kalıcılık depolarıyla birlikte kendi sınırlı bağlamlarına sahip olmaya devam etmelidir.</span><span class="sxs-lookup"><span data-stu-id="75cff-188">At a minimum, individual web applications should strive to be their own bounded context, with their own persistence store for their business model, rather than sharing a database with other applications.</span></span> <span data-ttu-id="75cff-189">Sınırlanmış bağlamlar arasındaki iletişim, paylaşılan bir veritabanı yerine, iş mantığı ve olayların gerçekleşen değişikliklere yanıt olarak gerçekleşmesini sağlayan, paylaşılan bir veritabanı yerine programlı arabirimler aracılığıyla oluşur.</span><span class="sxs-lookup"><span data-stu-id="75cff-189">Communication between bounded contexts occurs through programmatic interfaces, rather than through a shared database, which allows for business logic and events to take place in response to changes that take place.</span></span> <span data-ttu-id="75cff-190">Sınırlanmış bağlamlar, mikro hizmetlere yakın bir şekilde eşlenir ve ayrıca kendi kendine sınırlanmış bağlamlar olarak ideal şekilde uygulanır.</span><span class="sxs-lookup"><span data-stu-id="75cff-190">Bounded contexts map closely to microservices, which also are ideally implemented as their own individual bounded contexts.</span></span>
