    {

        // ...

        var orderItem = new OrderItem(productId, productName, unitPrice, discount, pictureUrl, units);

        _orderItems.Add(orderItem);

    }
    // ...

    // ...

## <a name="encapsulate-data-in-the-domain-entities"></a><span data-ttu-id="2284d-147">Etki Alanı Varlıklarındaki verileri kapsülle</span><span class="sxs-lookup"><span data-stu-id="2284d-147">Encapsulate data in the Domain Entities</span></span>

<span data-ttu-id="2284d-148">Varlık modellerinde sık karşılaşılan bir sorun, koleksiyon gezinti özelliklerini genel olarak erişilebilir liste türleri olarak ortaya çıkarmalarıdır.</span><span class="sxs-lookup"><span data-stu-id="2284d-148">A common problem in entity models is that they expose collection navigation properties as publicly accessible list types.</span></span> <span data-ttu-id="2284d-149">Bu, herhangi bir ortak geliştiricinin, koleksiyonla ilgili önemli iş kurallarını atlayarak nesneyi geçersiz bir durumda bırakabilecek bu koleksiyon türlerinin içeriğini işlemesine olanak tanır.</span><span class="sxs-lookup"><span data-stu-id="2284d-149">This allows any collaborator developer to manipulate the contents of these collection types, which may bypass important business rules related to the collection, possibly leaving the object in an invalid state.</span></span> <span data-ttu-id="2284d-150">Bunun çözümü, ilgili koleksiyonlara salt okunur erişimi ortaya çıkarmak ve istemcilerin bunları işleme yollarını tanımlayan yöntemler açıkça sağlamaktır.</span><span class="sxs-lookup"><span data-stu-id="2284d-150">The solution to this is to expose read-only access to related collections and explicitly provide methods that define ways in which clients can manipulate them.</span></span>

<span data-ttu-id="2284d-151">Önceki kodda, birçok özniteliklerin salt okunur veya özel olduğunu ve yalnızca sınıf yöntemleri tarafından güncelleştirilebilen olduğunu unutmayın, bu nedenle herhangi bir güncelleştirme iş etki alanı değişkenlerini ve sınıf yöntemleri içinde belirtilen mantığı dikkate alır.</span><span class="sxs-lookup"><span data-stu-id="2284d-151">In the previous code, note that many attributes are read-only or private and are only updatable by the class methods, so any update considers business domain invariants and logic specified within the class methods.</span></span>

<span data-ttu-id="2284d-152">Örneğin, DDD desenleri aşağıdaki aşağıdaki \*\*leri yapmak gerekir *not* \*\* herhangi bir komut işleyicisi yöntemi veya uygulama katmanı sınıfından (aslında, bunu yapmak için imkansız olmalıdır):</span><span class="sxs-lookup"><span data-stu-id="2284d-152">For example, following DDD patterns, **you should *not* do the following** from any command handler method or application layer class (actually, it should be impossible for you to do so):</span></span>

//... (WRONG) Some code with business logic out of the domain classes ...

<span data-ttu-id="2284d-153">Bu durumda, Ekle yöntemi yalnızca Sipariş Öğeleri koleksiyonuna doğrudan erişim ile veri eklemek için bir işlemdir.</span><span class="sxs-lookup"><span data-stu-id="2284d-153">In this case, the Add method is purely an operation to add data, with direct access to the OrderItems collection.</span></span> <span data-ttu-id="2284d-154">Bu nedenle, alt varlıklarla bu işlemle ilgili etki alanı mantığının, kurallarının veya doğrulamalarının çoğu uygulama katmanına (komut işleyicileri ve Web API denetleyicileri) yayılır.</span><span class="sxs-lookup"><span data-stu-id="2284d-154">Therefore, most of the domain logic, rules, or validations related to that operation with the child entities will be spread across the application layer (command handlers and Web API controllers).</span></span>

<span data-ttu-id="2284d-155">Toplam kökün etrafından dolaşırsanız, toplam kök değişmezlerini, geçerliliğini veya tutarlılığını garanti edemez.</span><span class="sxs-lookup"><span data-stu-id="2284d-155">If you go around the aggregate root, the aggregate root cannot guarantee its invariants, its validity, or its consistency.</span></span> <span data-ttu-id="2284d-156">Sonunda spagetti kodu veya işlem komut dosyası kodu olacaktır.</span><span class="sxs-lookup"><span data-stu-id="2284d-156">Eventually you will have spaghetti code or transactional script code.</span></span>

<span data-ttu-id="2284d-157">DDD modellerini izlemek için, varlıkların herhangi bir varlık özelliğinde ortak ayarlayıcıları olmamalıdır.</span><span class="sxs-lookup"><span data-stu-id="2284d-157">To follow DDD patterns, entities must not have public setters in any entity property.</span></span> <span data-ttu-id="2284d-158">Bir varlıktaki değişiklikler, varlıkta gerçekleştirdikleri değişiklik hakkında açık bir şekilde her yerde bulunan bir dille açık yöntemlerle yönlendirilmelidir.</span><span class="sxs-lookup"><span data-stu-id="2284d-158">Changes in an entity should be driven by explicit methods with explicit ubiquitous language about the change they are performing in the entity.</span></span>

<span data-ttu-id="2284d-159">Ayrıca, varlık içindeki koleksiyonlar (sipariş öğeleri gibi) salt okunur özellikler olmalıdır (AsReadOnly yöntemi daha sonra açıklanır).</span><span class="sxs-lookup"><span data-stu-id="2284d-159">Furthermore, collections within the entity (like the order items) should be read-only properties (the AsReadOnly method explained later).</span></span> <span data-ttu-id="2284d-160">Yalnızca toplu kök sınıf yöntemleri veya alt varlık yöntemleri içinden güncelleştirebilirsiniz.</span><span class="sxs-lookup"><span data-stu-id="2284d-160">You should be able to update it only from within the aggregate root class methods or the child entity methods.</span></span>

<span data-ttu-id="2284d-161">Sipariş toplu kökü için kodda görebileceğiniz gibi, varlığın verilerine veya alt varlıklarına karşı herhangi bir işlemin varlık sınıfındaki yöntemlerle gerçekleştirilebilmesi için tüm ayarlayıcıların özel veya en azından salt okunur dıştan olması gerekir.</span><span class="sxs-lookup"><span data-stu-id="2284d-161">As you can see in the code for the Order aggregate root, all setters should be private or at least read-only externally, so that any operation against the entity’s data or its child entities has to be performed through methods in the entity class.</span></span> <span data-ttu-id="2284d-162">Bu, işlem komut dosyası kodunu uygulamak yerine tutarlılığı denetimli ve nesne yönelimli bir şekilde korur.</span><span class="sxs-lookup"><span data-stu-id="2284d-162">This maintains consistency in a controlled and object-oriented way instead of implementing transactional script code.</span></span>

<span data-ttu-id="2284d-163">Aşağıdaki kod snippet Sipariş toplamına bir Sipariş Öğesi nesnesi ekleme görevini kodlamak için uygun yolu gösterir.</span><span class="sxs-lookup"><span data-stu-id="2284d-163">The following code snippet shows the proper way to code the task of adding an OrderItem object to the Order aggregate.</span></span>

<span data-ttu-id="2284d-164">Bu snippet'te, Bir OrderItem nesnesinin oluşturulmasıyla ilgili doğrulamaların veya mantığın çoğu, AddOrderItem yönteminde Sipariş toplu kökü nün denetimi altında olacaktır- özellikle doğrulamalar ve toplamdaki diğer öğelerle ilgili mantık.</span><span class="sxs-lookup"><span data-stu-id="2284d-164">In this snippet, most of the validations or logic related to the creation of an OrderItem object will be under the control of the Order aggregate root—in the AddOrderItem method—especially validations and logic related to other elements in the aggregate.</span></span> <span data-ttu-id="2284d-165">Örneğin, AddOrderItem'e yapılan birden çok arama sonucunda aynı ürün öğesini alabilirsiniz.</span><span class="sxs-lookup"><span data-stu-id="2284d-165">For instance, you might get the same product item as the result of multiple calls to AddOrderItem.</span></span> <span data-ttu-id="2284d-166">Bu yöntemde, ürün öğelerini inceleyebilir ve aynı ürün öğelerini birkaç birimiçeren tek bir OrderItem nesnesi içinde birleştirebilirsiniz.</span><span class="sxs-lookup"><span data-stu-id="2284d-166">In that method, you could examine the product items and consolidate the same product items into a single OrderItem object with several units.</span></span> <span data-ttu-id="2284d-167">Ayrıca, farklı indirim tutarları varsa ancak ürün kimliği aynıysa, büyük olasılıkla daha yüksek iskonto uygularsınız.</span><span class="sxs-lookup"><span data-stu-id="2284d-167">Additionally, if there are different discount amounts but the product ID is the same, you would likely apply the higher discount.</span></span> <span data-ttu-id="2284d-168">Bu ilke, OrderItem nesnesi için başka bir etki alanı mantığı için geçerlidir.</span><span class="sxs-lookup"><span data-stu-id="2284d-168">This principle applies to any other domain logic for the OrderItem object.</span></span>

<span data-ttu-id="2284d-169">Buna ek olarak, yeni OrderItem(params) işlemi de denetlenecek ve AddOrderItem yöntemi tarafından Sipariş toplam kökünden gerçekleştirilir.</span><span class="sxs-lookup"><span data-stu-id="2284d-169">In addition, the new OrderItem(params) operation will also be controlled and performed by the AddOrderItem method from the Order aggregate root.</span></span> <span data-ttu-id="2284d-170">Bu nedenle, bu işlemle ilgili mantık veya doğrulamaların çoğu (özellikle diğer alt varlıklar arasındaki tutarlılığı etkileyen her şey) toplam kök içinde tek bir yerde olacaktır.</span><span class="sxs-lookup"><span data-stu-id="2284d-170">Therefore, most of the logic or validations related to that operation (especially anything that impacts the consistency between other child entities) will be in a single place within the aggregate root.</span></span> <span data-ttu-id="2284d-171">Bu toplam kök deseni nihai amacıdır.</span><span class="sxs-lookup"><span data-stu-id="2284d-171">That is the ultimate purpose of the aggregate root pattern.</span></span>

<span data-ttu-id="2284d-172">Varlık Framework Core 1.1 veya daha sonra kullandığınızda, özellikler ek olarak [alanlara eşleme](https://docs.microsoft.com/ef/core/modeling/backing-field) sağlar, çünkü bir DDD varlık daha iyi ifade edilebilir.</span><span class="sxs-lookup"><span data-stu-id="2284d-172">When you use Entity Framework Core 1.1 or later, a DDD entity can be better expressed because it allows [mapping to fields](https://docs.microsoft.com/ef/core/modeling/backing-field) in addition to properties.</span></span> <span data-ttu-id="2284d-173">Bu, alt varlıkların veya değer nesnelerinin koleksiyonlarını korurken yararlıdır.</span><span class="sxs-lookup"><span data-stu-id="2284d-173">This is useful when protecting collections of child entities or value objects.</span></span> <span data-ttu-id="2284d-174">Bu geliştirme yle, özellikler yerine basit özel alanlar kullanabilir ve ortak yöntemlerle alan koleksiyonuna herhangi bir güncelleştirme uygulayabilir ve AsReadOnly yöntemi yle salt okunur erişim sağlayabilirsiniz.</span><span class="sxs-lookup"><span data-stu-id="2284d-174">With this enhancement, you can use simple private fields instead of properties and you can implement any update to the field collection in public methods and provide read-only access through the AsReadOnly method.</span></span>
