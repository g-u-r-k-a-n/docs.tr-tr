# <a name="abstractions-abstract-types-and-interfaces"></a><span data-ttu-id="25200-102">Soyutlamalar (Soyut Türler ve Arabirimler)</span><span class="sxs-lookup"><span data-stu-id="25200-102">Abstractions (Abstract Types and Interfaces)</span></span>
<span data-ttu-id="25200-103">Soyutlama, sözleşmeyi tanımlayan, ancak sözleşmenin tam bir uygulamasını sağlamayan bir türdür.</span><span class="sxs-lookup"><span data-stu-id="25200-103">An abstraction is a type that describes a contract but does not provide a full implementation of the contract.</span></span> <span data-ttu-id="25200-104">Soyutlamalar genellikle soyut sınıflar veya arabirimler olarak uygulanır ve sözleşmeyi uygulayan türlerin gereken semantiğini açıklayan, iyi tanımlanmış bir başvuru belgeleri kümesiyle gelir.</span><span class="sxs-lookup"><span data-stu-id="25200-104">Abstractions are usually implemented as abstract classes or interfaces, and they come with a well-defined set of reference documentation describing the required semantics of the types implementing the contract.</span></span> <span data-ttu-id="25200-105">.NET Framework en önemli soyutlamalar <xref:System.IO.Stream>, <xref:System.Collections.Generic.IEnumerable%601>ve <xref:System.Object>içerir.</span><span class="sxs-lookup"><span data-stu-id="25200-105">Some of the most important abstractions in the .NET Framework include <xref:System.IO.Stream>, <xref:System.Collections.Generic.IEnumerable%601>, and <xref:System.Object>.</span></span>

 <span data-ttu-id="25200-106">Bir soyutlamayı destekleyen somut bir tür uygulayarak ve soyutlamayı kullanan (üzerinde çalışan) çerçeve API 'Leriyle bu somut türü kullanarak çerçeveleri genişletebilirsiniz.</span><span class="sxs-lookup"><span data-stu-id="25200-106">You can extend frameworks by implementing a concrete type that supports the contract of an abstraction and using this concrete type with framework APIs consuming (operating on) the abstraction.</span></span>

 <span data-ttu-id="25200-107">Zamanı test etmek mümkün olan anlamlı ve yararlı bir soyutlama tasarlamak çok zordur.</span><span class="sxs-lookup"><span data-stu-id="25200-107">A meaningful and useful abstraction that is able to withstand the test of time is very difficult to design.</span></span> <span data-ttu-id="25200-108">Ana zorluk, daha fazla ve daha az olmayan, doğru üye kümesini alıyor.</span><span class="sxs-lookup"><span data-stu-id="25200-108">The main difficulty is getting the right set of members, no more and no fewer.</span></span> <span data-ttu-id="25200-109">Soyutlama çok fazla üyeye sahipse, uygulanması zor veya imkansız olur.</span><span class="sxs-lookup"><span data-stu-id="25200-109">If an abstraction has too many members, it becomes difficult or even impossible to implement.</span></span> <span data-ttu-id="25200-110">Taahhüt edilen işlevsellik için çok az üye varsa, bu çok sayıda ilginç senaryo için kullanılamaz hale gelir.</span><span class="sxs-lookup"><span data-stu-id="25200-110">If it has too few members for the promised functionality, it becomes useless in many interesting scenarios.</span></span>

 <span data-ttu-id="25200-111">Çerçevede çok fazla soyut olan soyutlamalar ayrıca Framework kullanılabilirliğini olumsuz etkiler.</span><span class="sxs-lookup"><span data-stu-id="25200-111">Too many abstractions in a framework also negatively affect usability of the framework.</span></span> <span data-ttu-id="25200-112">Somut uygulamaların ve soyutlama üzerinde çalışan API 'lerin daha büyük resmine ne kadar uygun olduğunu anlamak zorunda kalmadan bir soyutlamayı anlamak oldukça zordur.</span><span class="sxs-lookup"><span data-stu-id="25200-112">It is often quite difficult to understand an abstraction without understanding how it fits into the larger picture of the concrete implementations and the APIs operating on the abstraction.</span></span> <span data-ttu-id="25200-113">Ayrıca, soyutlamalar ve üyelerinin adları soyut hale gelir, bu da genellikle kullanımının daha geniş bir bağlamını anlamak zorunda kalmadan şifreli ve unapproachable.</span><span class="sxs-lookup"><span data-stu-id="25200-113">Also, names of abstractions and their members are necessarily abstract, which often makes them cryptic and unapproachable without first understanding the broader context of their usage.</span></span>

 <span data-ttu-id="25200-114">Ancak soyutlamalar, diğer genişletilebilirlik mekanizmalarının sıklıkla eşleşemesinden çok güçlü bir genişletilebilirlik sağlar.</span><span class="sxs-lookup"><span data-stu-id="25200-114">However, abstractions provide extremely powerful extensibility that the other extensibility mechanisms cannot often match.</span></span> <span data-ttu-id="25200-115">Eklentiler, denetim (IOC), işlem hatları vb. gibi birçok mimari deseninden oluşur.</span><span class="sxs-lookup"><span data-stu-id="25200-115">They are at the core of many architectural patterns, such as plug-ins, inversion of control (IoC), pipelines, and so on.</span></span> <span data-ttu-id="25200-116">Çerçeveler için test edilebilirlik için de oldukça önemlidir.</span><span class="sxs-lookup"><span data-stu-id="25200-116">They are also extremely important for testability of frameworks.</span></span> <span data-ttu-id="25200-117">İyi soyutlamalar, birim testi amacıyla ağır bağımlılıkların yerine koymasını mümkün kılar.</span><span class="sxs-lookup"><span data-stu-id="25200-117">Good abstractions make it possible to stub out heavy dependencies for the purpose of unit testing.</span></span> <span data-ttu-id="25200-118">Özet olarak, modern nesne yönelimli çerçevelerin zenginleştirmesinden sonra soyutlamalar sorumludur.</span><span class="sxs-lookup"><span data-stu-id="25200-118">In summary, abstractions are responsible for the sought-after richness of the modern object-oriented frameworks.</span></span>

 <span data-ttu-id="25200-119">❌ soyutlamaları kullanan çeşitli somut uygulamalar ve API 'Ler geliştirilirken test olmadıkları müddetçe soyutlamalar sağlamaz.</span><span class="sxs-lookup"><span data-stu-id="25200-119">❌ DO NOT provide abstractions unless they are tested by developing several concrete implementations and APIs consuming the abstractions.</span></span>

 <span data-ttu-id="25200-120">bir soyutlama tasarlarken bir soyut sınıf ve arabirim arasında dikkatli bir şekilde seçim YAPıN. ✔️</span><span class="sxs-lookup"><span data-stu-id="25200-120">✔️ DO choose carefully between an abstract class and an interface when designing an abstraction.</span></span>

 <span data-ttu-id="25200-121">✔️ Soyut uygulamaların somut uygulamaları için başvuru testleri sağlamayı düşünün.</span><span class="sxs-lookup"><span data-stu-id="25200-121">✔️ CONSIDER providing reference tests for concrete implementations of abstractions.</span></span> <span data-ttu-id="25200-122">Bu tür testler, kullanıcıların, sözleşmesinin sözleşmeyi doğru şekilde uygulayıp uygulamadığını test etmelerini sağlar.</span><span class="sxs-lookup"><span data-stu-id="25200-122">Such tests should allow users to test whether their implementations correctly implement the contract.</span></span>
