<span data-ttu-id="43a47-104">Bulutta yerel uygulamalar geleneksel tek örnekli tek parçalı uygulamalar 'dan daha fazla çalışan çok sayıda hizmet içerir.</span><span class="sxs-lookup"><span data-stu-id="43a47-104">Cloud-native applications involve many more running services than traditional single-instance monolithic apps.</span></span> <span data-ttu-id="43a47-105">Birbirine bağlı onlarca hizmet veren hizmetlerin yapılandırma ayarlarını yönetmek zor olabilir, bu nedenle merkezi yapılandırma depoları genellikle dağıtılmış uygulamalar için uygulanır.</span><span class="sxs-lookup"><span data-stu-id="43a47-105">Managing configuration settings for dozens of interdependent services can be challenging, which is why centralized configuration stores are often implemented for distributed applications.</span></span>

<span data-ttu-id="43a47-106">[Bölüm 1](introduction.md)' de anlatıldığı gibi, on Iki öğeli uygulama önerileri, kod ve yapılandırma arasında katı ayrım gerektirir.</span><span class="sxs-lookup"><span data-stu-id="43a47-106">As discussed in [Chapter 1](introduction.md), the Twelve-Factor App recommendations require strict separation between code and configuration.</span></span> <span data-ttu-id="43a47-107">Bu, yapılandırma ayarlarının sabitler olarak depolanması veya koddaki değişmez değerlerin bir ihlalin olması anlamına gelir.</span><span class="sxs-lookup"><span data-stu-id="43a47-107">This means storing configuration settings as constants or literal values in code is a violation.</span></span> <span data-ttu-id="43a47-108">Geliştirme, test, hazırlama ve üretim dahil olmak üzere birden çok ortamda aynı kodun kullanılması gerektiğinden bu öneri vardır.</span><span class="sxs-lookup"><span data-stu-id="43a47-108">This recommendation exists because the same code should be used across multiple environments, including development, testing, staging, and production.</span></span> <span data-ttu-id="43a47-109">Ancak, yapılandırma değerleri bu ortamların her biri arasında farklılık gösterebilir.</span><span class="sxs-lookup"><span data-stu-id="43a47-109">However, configuration values are likely to vary between each of these environments.</span></span> <span data-ttu-id="43a47-110">Bu nedenle, yapılandırma değerleri ortamın kendisinde depolanmalıdır veya ortamın kimlik bilgilerini merkezi bir yapılandırma deposuna depolaması gerekir.</span><span class="sxs-lookup"><span data-stu-id="43a47-110">So, configuration values should be stored in the environment itself, or the environment should store the credentials to a centralized configuration store.</span></span>

<span data-ttu-id="43a47-111">EShopOnContainers uygulaması, her mikro hizmetle yerel uygulama ayarları dosyalarını içerir.</span><span class="sxs-lookup"><span data-stu-id="43a47-111">The eShopOnContainers application includes local application settings files with each microservice.</span></span> <span data-ttu-id="43a47-112">Bu dosyalar kaynak denetimine iade edilir ancak bağlantı dizeleri veya API anahtarları gibi üretim sırları dahil değildir.</span><span class="sxs-lookup"><span data-stu-id="43a47-112">These files are checked into source control but don't include production secrets such as connection strings or API keys.</span></span> <span data-ttu-id="43a47-113">Üretimde, hizmet başına ortam değişkenleriyle bireysel ayarların üzerine yazılabilir.</span><span class="sxs-lookup"><span data-stu-id="43a47-113">In production, individual settings may be overwritten with per-service environment variables.</span></span> <span data-ttu-id="43a47-114">Bu, barındırılan uygulamalar için yaygın bir uygulamadır, ancak merkezi bir yapılandırma deposu sağlamaz.</span><span class="sxs-lookup"><span data-stu-id="43a47-114">This is a common practice for hosted applications, but doesn't provide a central configuration store.</span></span> <span data-ttu-id="43a47-115">Yapılandırma ayarlarının merkezi yönetimini desteklemek için her mikro hizmet, yerel ayarları veya Azure Key Vault ayarları kullanımı arasında geçiş yapmak için bir ayar içerir.</span><span class="sxs-lookup"><span data-stu-id="43a47-115">To support centralized management of configuration settings, each microservice includes a setting to toggle between its use of local settings or Azure Key Vault settings.</span></span>

## <a name="azure-key-vault"></a><span data-ttu-id="43a47-116">Azure anahtar kasası</span><span class="sxs-lookup"><span data-stu-id="43a47-116">Azure Key Vault</span></span>

<span data-ttu-id="43a47-117">Azure Key Vault, belirteçlerin, parolaların, sertifikaların, API anahtarlarının ve diğer gizli parolaların güvenli bir şekilde depolanmasını sağlar.</span><span class="sxs-lookup"><span data-stu-id="43a47-117">Azure Key Vault provides secure storage of tokens, passwords, certificates, API keys, and other sensitive secrets.</span></span> <span data-ttu-id="43a47-118">Key Vault erişim, eShopOnContainers mikro hizmetleri için bir ClientID/ClientSecret bileşiminin kullanılması gereken, uygun bir arayan kimlik doğrulaması ve yetkilendirme gerektirir.</span><span class="sxs-lookup"><span data-stu-id="43a47-118">Access to Key Vault requires proper caller authentication and authorization, which in the case of the eShopOnContainers microservices means the use of a ClientId/ClientSecret combination.</span></span> <span data-ttu-id="43a47-119">Bu kimlik bilgilerini kaynak denetimine iade etmeyin, ancak bunun yerine uygulamanın ortamında ayarlayın.</span><span class="sxs-lookup"><span data-stu-id="43a47-119">Don't check in these credentials into source control, but instead set in the application's environment.</span></span> <span data-ttu-id="43a47-120">AKS 'ten Key Vault doğrudan erişim, [Key Vault FlexVolume](https://github.com/Azure/kubernetes-keyvault-flexvol)kullanılarak elde edilebilir.</span><span class="sxs-lookup"><span data-stu-id="43a47-120">Direct access to Key Vault from AKS can be achieved using [Key Vault FlexVolume](https://github.com/Azure/kubernetes-keyvault-flexvol).</span></span>
