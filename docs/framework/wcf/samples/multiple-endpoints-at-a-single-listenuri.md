## <a name="service"></a><span data-ttu-id="74721-115">Hizmet</span><span class="sxs-lookup"><span data-stu-id="74721-115">Service</span></span>

        binding="wsHttpBinding"  
        contract="Microsoft.ServiceModel.Samples.ICalculator"   
        listenUri="http://localhost/servicemodelsamples/service.svc" />  
<endpoint address="urn:Stuff"  
        binding="wsHttpBinding"  
        contract="Microsoft.ServiceModel.Samples.IEcho"   
        listenUri="http://localhost/servicemodelsamples/service.svc" />  
<endpoint address="urn:OtherEcho"  
        binding="wsHttpBinding"  
        contract="Microsoft.ServiceModel.Samples.IEcho"   
        listenUri="http://localhost/servicemodelsamples/service.svc" />

 <span data-ttu-id="74721-118">Üç uç noktanın tümü aynı `ListenUri` barındırılır ve aynı `binding` uç noktaların aynı `ListenUri` aynı bağlama sahip olması gerekir, çünkü makinede o fiziksel adresteki iletileri dinleyen tek bir kanal yığınını paylaşıyor.</span><span class="sxs-lookup"><span data-stu-id="74721-118">All three endpoints are hosted at the same `ListenUri` and use the same `binding` - endpoints at the same `ListenUri` must have the same binding, because they are sharing a single channel stack that listens for messages at that physical address on the machine.</span></span> <span data-ttu-id="74721-119">Her uç noktanın `address` URN olur; genellikle adresler fiziksel konumları temsil ediyorsa, aslında adres, bu örnekte gösterildiği gibi eşleştirme ve filtreleme amaçları için kullanıldığından adres herhangi bir tür URI olabilir.</span><span class="sxs-lookup"><span data-stu-id="74721-119">The `address` of each endpoint is a URN; though typically addresses represent physical locations, in fact the address can be any kind of URI, because the address is used for matching and filtering purposes as is demonstrated in this sample.</span></span>  
  
 <span data-ttu-id="74721-120">Üç uç noktanın tümü aynı `ListenUri`paylaştığından, bir ileti buraya ulaştığında, Windows Communication Foundation (WCF) iletinin hangi uç noktaya ait olduğunu karar <span data-ttu-id="74721-122">Adres filtresi, SOAP iletisinin `To` hizmet uç noktasının adresiyle eşleştirir.</span><span class="sxs-lookup"><span data-stu-id="74721-122">The address filter matches <span data-ttu-id="74721-124">Sözleşme filtresi, belirli bir sözleşmenin işlemleriyle ilişkili eylemlerle eşleşir.</span><span class="sxs-lookup"><span data-stu-id="74721-124">The contract filter ## <a name="client"></a><span data-ttu-id="74721-130">İstemci</span><span class="sxs-lookup"><span data-stu-id="74721-130">Client</span></span>

 <span data-ttu-id="74721-136">Her zamanki gibi, adres, Svcutil. exe tarafından oluşturulan istemci yapılandırma dosyasından gelir.</span><span class="sxs-lookup"><span data-stu-id="74721-136">As usual, the address comes from the client configuration file, which was generated by Svcutil.exe.</span></span> <span data-ttu-id="74721-137">`Via` (hizmetin `ListenUri` karşılık gelen), hizmetin meta verilerinde görünmez ve bu nedenle bu bilgilerin istemciye bant dışı (hizmetin meta veri adresi gibi) bildirilmesi gerekir.</span><span class="sxs-lookup"><span data-stu-id="74721-137">The `Via` (which corresponds to the `ListenUri` of the service) does not appear in the service's metadata and so this information must be communicated to the client out-of-band (just like the service's metadata address).</span></span>  
  
 <span data-ttu-id="74721-138">Bu örnekteki istemci, her bir sunucunun üç uygulama uç noktasına ileti gönderir ve bunların hepsi aynı `Via`sahip olsa da üç uç nokta ile iletişim kurabildiğini gösterir.</span><span class="sxs-lookup"><span data-stu-id="74721-138">The client in this sample sends messages to each of the server's three application endpoints, to demonstrate that it can communicate with (and differentiate) all three endpoints, even though they all have the same `Via`.</span></span>
