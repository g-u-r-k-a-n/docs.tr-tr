# <a name="managing-permissions-with-stored-procedures-in-sql-server"></a><span data-ttu-id="43347-103">SQL Server'da Saklı Yordam İzinlerini Yönetme</span><span class="sxs-lookup"><span data-stu-id="43347-103">Managing Permissions with Stored Procedures in SQL Server</span></span>

<span data-ttu-id="43347-104">Veritabanınızda birden çok savunma hattı oluşturmanın bir yöntemi, saklı yordamları veya Kullanıcı tanımlı işlevleri kullanarak tüm veri erişimini uygulamaktır.</span><span class="sxs-lookup"><span data-stu-id="43347-104">One method of creating multiple lines of defense around your database is to implement all data access using stored procedures or user-defined functions.</span></span> <span data-ttu-id="43347-105">Tablolar gibi temeldeki nesneler için tüm izinleri iptal eder veya reddeder ve saklı yordamlarda yürütme izinleri verir.</span><span class="sxs-lookup"><span data-stu-id="43347-105">You revoke or deny all permissions to underlying objects, such as tables, and grant EXECUTE permissions on stored procedures.</span></span> <span data-ttu-id="43347-106">Bu, veri ve veritabanı nesneleriniz etrafında etkin bir güvenlik çevresi oluşturur.</span><span class="sxs-lookup"><span data-stu-id="43347-106">This effectively creates a security perimeter around your data and database objects.</span></span>  
  
## <a name="stored-procedure-benefits"></a><span data-ttu-id="43347-107">Saklı yordam avantajları</span><span class="sxs-lookup"><span data-stu-id="43347-107">Stored Procedure Benefits</span></span>  

 <span data-ttu-id="43347-108">Saklı yordamlar aşağıdaki avantajlara sahiptir:</span><span class="sxs-lookup"><span data-stu-id="43347-108">Stored procedures have the following benefits:</span></span>  
  
- <span data-ttu-id="43347-109">Veri mantığı ve iş kuralları, kullanıcıların veri ve nesnelere yalnızca geliştiricilerin ve veritabanı yöneticilerinin tarafından istenen şekilde erişebilmeleri için kapsüllenebilir.</span><span class="sxs-lookup"><span data-stu-id="43347-109">Data logic and business rules can be encapsulated so that users can access data and objects only in ways that developers and database administrators intend.</span></span>  
  
- <span data-ttu-id="43347-110">Tüm Kullanıcı girişlerini doğrulayan parametreli saklı yordamlar, SQL ekleme saldırılarını sağlamak için kullanılabilir.</span><span class="sxs-lookup"><span data-stu-id="43347-110">Parameterized stored procedures that validate all user input can be used to thwart SQL injection attacks.</span></span> <span data-ttu-id="43347-111">Dinamik SQL kullanıyorsanız, komutlarınızı parametreleştirilediğinizden ve parametre değerlerini hiçbir şekilde doğrudan bir sorgu dizesine eklemeyin.</span><span class="sxs-lookup"><span data-stu-id="43347-111">If you use dynamic SQL, be sure to parameterize your commands, and never include parameter values directly into a query string.</span></span>  
  
- <span data-ttu-id="43347-112">Geçici sorgulara ve veri değişikliklerine izin verilmez.</span><span class="sxs-lookup"><span data-stu-id="43347-112">Ad hoc queries and data modifications can be disallowed.</span></span> <span data-ttu-id="43347-113">Bu, kullanıcıların kötü amaçlı veya istenmeden veri yok etmesini ya da sunucu veya ağ üzerindeki performansa zarar veren sorguları yürütmesini engeller.</span><span class="sxs-lookup"><span data-stu-id="43347-113">This prevents users from maliciously or inadvertently destroying data or executing queries that impair performance on the server or the network.</span></span>  
  
- <span data-ttu-id="43347-114">Hatalar, doğrudan istemci uygulamalarına geçirilmeden yordam kodunda işlenebilir.</span><span class="sxs-lookup"><span data-stu-id="43347-114">Errors can be handled in procedure code without being passed directly to client applications.</span></span> <span data-ttu-id="43347-115">Bu, bir yoklama saldırısında yardımcı olabilecek hata iletilerinin döndürülmesini önler.</span><span class="sxs-lookup"><span data-stu-id="43347-115">This prevents error messages from being returned that could aid in a probing attack.</span></span> <span data-ttu-id="43347-116">Hataları günlüğe kaydedin ve sunucuda işleyin.</span><span class="sxs-lookup"><span data-stu-id="43347-116">Log errors and handle them on the server.</span></span>  
  
- <span data-ttu-id="43347-117">Saklı yordamlar bir kez yazılabilir ve birçok uygulama tarafından erişilebilir.</span><span class="sxs-lookup"><span data-stu-id="43347-117">Stored procedures can be written once, and accessed by many applications.</span></span>  
  
- <span data-ttu-id="43347-118">İstemci uygulamaların, temel alınan veri yapıları hakkında herhangi bir şeyi bilmeleri gerekmez.</span><span class="sxs-lookup"><span data-stu-id="43347-118">Client applications do not need to know anything about the underlying data structures.</span></span> <span data-ttu-id="43347-119">Saklı yordam kodu, değişiklikler parametre listelerini veya döndürülen veri türlerini etkilemediği sürece istemci uygulamalarında değişiklik gerektirmeden değiştirilebilir.</span><span class="sxs-lookup"><span data-stu-id="43347-119">Stored procedure code can be changed without requiring changes in client applications as long as the changes do not affect parameter lists or returned data types.</span></span>  
  
- <span data-ttu-id="43347-120">Saklı yordamlar, birden çok işlemi tek bir yordam çağrısında birleştirerek ağ trafiğini azaltabilir.</span><span class="sxs-lookup"><span data-stu-id="43347-120">Stored procedures can reduce network traffic by combining multiple operations into one procedure call.</span></span>  
  
## <a name="stored-procedure-execution"></a><span data-ttu-id="43347-121">Saklı yordam yürütme</span><span class="sxs-lookup"><span data-stu-id="43347-121">Stored Procedure Execution</span></span>  

 <span data-ttu-id="43347-122">Saklı yordamlar, kullanıcıların veritabanına erişim sağlamak için sahiplik zincirinden yararlanır ve böylece kullanıcılar veritabanı nesnelerine erişim için açık izne sahip olmaları gerekmez.</span><span class="sxs-lookup"><span data-stu-id="43347-122">Stored procedures take advantage of ownership chaining to provide access to data so that users do not need to have explicit permission to access database objects.</span></span> <span data-ttu-id="43347-123">Bir sahiplik zinciri birbirlerine her ardışık olarak erişen nesneler aynı kullanıcıya ait olduğunda oluşur.</span><span class="sxs-lookup"><span data-stu-id="43347-123">An ownership chain exists when objects that access each other sequentially are owned by the same user.</span></span> <span data-ttu-id="43347-124">Örneğin, saklı yordam diğer saklı yordamları çağırabilir veya saklı yordam birden fazla tabloya erişebilir.</span><span class="sxs-lookup"><span data-stu-id="43347-124">For example, a stored procedure can call other stored procedures, or a stored procedure can access multiple tables.</span></span> <span data-ttu-id="43347-125">Yürütme zincirindeki tüm nesneler aynı sahibe sahip ise SQL Server, yalnızca arayan için yürütme iznini denetler, çağıranın diğer nesneler üzerindeki izinleri değildir.</span><span class="sxs-lookup"><span data-stu-id="43347-125">If all objects in the chain of execution have the same owner, then SQL Server only checks the EXECUTE permission for the caller, not the caller's permissions on other objects.</span></span> <span data-ttu-id="43347-126">Bu nedenle, yalnızca saklı yordamlarda yürütme izinleri vermeniz gerekir; temel alınan tablolardaki tüm izinleri iptal edebilir veya reddedebilirsiniz.</span><span class="sxs-lookup"><span data-stu-id="43347-126">Therefore you need to grant only EXECUTE permissions on stored procedures; you can revoke or deny all permissions on the underlying tables.</span></span>  
  
## <a name="best-practices"></a><span data-ttu-id="43347-127">En İyi Uygulamalar</span><span class="sxs-lookup"><span data-stu-id="43347-127">Best Practices</span></span>  

 <span data-ttu-id="43347-128">Saklı yordamları yazmak, uygulamanızı yeterince güvenli hale getirmek için yeterli değildir.</span><span class="sxs-lookup"><span data-stu-id="43347-128">Simply writing stored procedures isn't enough to adequately secure your application.</span></span> <span data-ttu-id="43347-129">Ayrıca, aşağıdaki olası güvenlik boşluklarını de göz önünde bulundurmanız gerekir.</span><span class="sxs-lookup"><span data-stu-id="43347-129">You should also consider the following potential security holes.</span></span>  
  
- <span data-ttu-id="43347-130">Verilere erişebilmek istediğiniz veritabanı rollerinin saklı yordamları üzerinde yürütme izinleri verin.</span><span class="sxs-lookup"><span data-stu-id="43347-130">Grant EXECUTE permissions on the stored procedures for database roles you want to be able to access the data.</span></span>  
  
- <span data-ttu-id="43347-131">Rol dahil olmak üzere veritabanındaki tüm roller ve kullanıcılar için temel alınan tabloların tüm izinlerini iptal edin veya reddedin `public` .</span><span class="sxs-lookup"><span data-stu-id="43347-131">Revoke or deny all permissions to the underlying tables for all roles and users in the database, including the `public` role.</span></span> <span data-ttu-id="43347-132">Tüm kullanıcılar izinleri herkese devralınır.</span><span class="sxs-lookup"><span data-stu-id="43347-132">All users inherit permissions from public.</span></span> <span data-ttu-id="43347-133">Bu nedenle, izinleri reddetme, `public` yalnızca sahiplerin ve `sysadmin` üyelerin erişimi olduğu anlamına gelir; diğer tüm kullanıcılar diğer rollerdeki üyelikle izinleri almayacaktır.</span><span class="sxs-lookup"><span data-stu-id="43347-133">Therefore denying permissions to `public` means that only owners and `sysadmin` members have access; all other users will be unable to inherit permissions from membership in other roles.</span></span>  
  
- <span data-ttu-id="43347-134">Veya rollerine Kullanıcı veya rol eklemeyin `sysadmin` `db_owner` .</span><span class="sxs-lookup"><span data-stu-id="43347-134">Do not add users or roles to the `sysadmin` or `db_owner` roles.</span></span> <span data-ttu-id="43347-135">Sistem yöneticileri ve veritabanı sahipleri, tüm veritabanı nesnelerine erişebilir.</span><span class="sxs-lookup"><span data-stu-id="43347-135">System administrators and database owners can access all database objects.</span></span>  
  
- <span data-ttu-id="43347-136">Hesabı devre dışı bırakın `guest` .</span><span class="sxs-lookup"><span data-stu-id="43347-136">Disable the `guest` account.</span></span> <span data-ttu-id="43347-137">Bu, anonim kullanıcıların veritabanına bağlanmasını engeller.</span><span class="sxs-lookup"><span data-stu-id="43347-137">This will prevent anonymous users from connecting to the database.</span></span> <span data-ttu-id="43347-138">Yeni veritabanlarında Konuk hesabı varsayılan olarak devre dışıdır.</span><span class="sxs-lookup"><span data-stu-id="43347-138">The guest account is disabled by default in new databases.</span></span>  
  
- <span data-ttu-id="43347-139">Hata işleme ve günlük hatalarını uygulayın.</span><span class="sxs-lookup"><span data-stu-id="43347-139">Implement error handling and log errors.</span></span>  
  
- <span data-ttu-id="43347-140">Tüm Kullanıcı girişlerini doğrulayan parametreli saklı yordamlar oluşturun.</span><span class="sxs-lookup"><span data-stu-id="43347-140">Create parameterized stored procedures that validate all user input.</span></span> <span data-ttu-id="43347-141">Tüm Kullanıcı girişlerini güvenilir değil olarak değerlendirin.</span><span class="sxs-lookup"><span data-stu-id="43347-141">Treat all user input as untrusted.</span></span>  
  
- <span data-ttu-id="43347-142">Mutlak gerekmedikçe dinamik SQL kullanmaktan kaçının.</span><span class="sxs-lookup"><span data-stu-id="43347-142">Avoid dynamic SQL unless absolutely necessary.</span></span> <span data-ttu-id="43347-143">Transact-SQL QUOTENAME () işlevini kullanarak bir dize değerini sınırlandırın ve giriş dizesindeki sınırlandırıcının herhangi bir oluşumunu kaçış.</span><span class="sxs-lookup"><span data-stu-id="43347-143">Use the Transact-SQL QUOTENAME() function to delimit a string value and escape any occurrence of the delimiter in the input string.</span></span>
