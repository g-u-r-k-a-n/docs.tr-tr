# <a name="creating-evolving-and-versioning-microservice-apis-and-contracts"></a><span data-ttu-id="b0f22-103">Mikro hizmet API’leri ve anlaşmaları oluşturma, geliştirme ve sürüm oluşturma</span><span class="sxs-lookup"><span data-stu-id="b0f22-103">Creating, evolving, and versioning microservice APIs and contracts</span></span>

<span data-ttu-id="b0f22-104">Microservice API, hizmet ve müşterileri arasında yapılan bir sözleşmedir.</span><span class="sxs-lookup"><span data-stu-id="b0f22-104">A microservice API is a contract between the service and its clients.</span></span> <span data-ttu-id="b0f22-105">Bir mikro hizmeti ancak API sözleşmesini bozmazsanız bağımsız olarak geliştirebilirsiniz, bu nedenle sözleşme çok önemlidir.</span><span class="sxs-lookup"><span data-stu-id="b0f22-105">You'll be able to evolve a microservice independently only if you do not break its API contract, which is why the contract is so important.</span></span> <span data-ttu-id="b0f22-106">Sözleşmeyi değiştirirseniz, istemci uygulamalarınızı veya API Ağ Geçidinizi etkiler.</span><span class="sxs-lookup"><span data-stu-id="b0f22-106">If you change the contract, it will impact your client applications or your API Gateway.</span></span>

<span data-ttu-id="b0f22-107">API tanımının yapısı, kullandığınız protokole bağlıdır.</span><span class="sxs-lookup"><span data-stu-id="b0f22-107">The nature of the API definition depends on which protocol you're using.</span></span> <span data-ttu-id="b0f22-108">Örneğin, ileti kullanıyorsanız [(AMQP](https://www.amqp.org/)gibi), API ileti türlerinden oluşur.</span><span class="sxs-lookup"><span data-stu-id="b0f22-108">For instance, if you're using messaging (like [AMQP](https://www.amqp.org/)), the API consists of the message types.</span></span> <span data-ttu-id="b0f22-109">HTTP ve RESTful hizmetlerini kullanıyorsanız, API URL'lerden ve istek ve yanıt JSON biçimlerinden oluşur.</span><span class="sxs-lookup"><span data-stu-id="b0f22-109">If you're using HTTP and RESTful services, the API consists of the URLs and the request and response JSON formats.</span></span>

<span data-ttu-id="b0f22-110">Ancak, ilk sözleşmeniz hakkında düşünceli olsanız bile, bir hizmet API'nin zaman içinde değişmesi gerekir.</span><span class="sxs-lookup"><span data-stu-id="b0f22-110">However, even if you're thoughtful about your initial contract, a service API will need to change over time.</span></span> <span data-ttu-id="b0f22-111">Bu durumda ve özellikle API'niz birden çok istemci uygulaması tarafından tüketilen genel bir API ise, genellikle tüm istemcileri yeni API sözleşmenize yükseltmeye zorlayamazsınız.</span><span class="sxs-lookup"><span data-stu-id="b0f22-111">When that happens—and especially if your API is a public API consumed by multiple client applications — you typically can't force all clients to upgrade to your new API contract.</span></span> <span data-ttu-id="b0f22-112">Genellikle bir hizmet sözleşmesinin hem eski hem de yeni sürümlerinin aynı anda çalıştırılacakları şekilde bir hizmetin yeni sürümlerini artımlı olarak dağıtmanız gerekir.</span><span class="sxs-lookup"><span data-stu-id="b0f22-112">You usually need to incrementally deploy new versions of a service in a way that both old and new versions of a service contract are running simultaneously.</span></span> <span data-ttu-id="b0f22-113">Bu nedenle, hizmet sürüm için bir strateji olması önemlidir.</span><span class="sxs-lookup"><span data-stu-id="b0f22-113">Therefore, it's important to have a strategy for your service versioning.</span></span>

<span data-ttu-id="b0f22-114">API değişiklikleri küçük olduğunda ( örneğin, API'nize öznitelikler veya parametreler eklerseniz, eski bir API kullanan istemciler, hizmetin yeni sürümüyle geçiş yapmalı ve bunlarla çalışmalıdır.</span><span class="sxs-lookup"><span data-stu-id="b0f22-114">When the API changes are small, like if you add attributes or parameters to your API, clients that use an older API should switch and work with the new version of the service.</span></span> <span data-ttu-id="b0f22-115">Gereken eksik öznitelikler için varsayılan değerler sağlayabilirsiniz ve istemciler herhangi bir ek yanıt özniteliklerini yok sayabilir.</span><span class="sxs-lookup"><span data-stu-id="b0f22-115">You might be able to provide default values for any missing attributes that are required, and the clients might be able to ignore any extra response attributes.</span></span>

<span data-ttu-id="b0f22-116">Ancak, bazen bir hizmet API'sinde önemli ve uyumsuz değişiklikler yapmanız gerekir.</span><span class="sxs-lookup"><span data-stu-id="b0f22-116">However, sometimes you need to make major and incompatible changes to a service API.</span></span> <span data-ttu-id="b0f22-117">İstemci uygulamalarını veya hizmetlerini hemen yeni sürüme yükseltmeye HTTP tabanlı bir mekanizma kullanıyorsanız, bir yaklaşım API sürüm numarasını URL'ye veya bir HTTP üstbilgisine katıştırmaktır.</span><span class="sxs-lookup"><span data-stu-id="b0f22-118">If you're using an HTTP-based mechanism such as REST, one approach is to embed the API version number in the URL or into an HTTP header.</span></span> <span data-ttu-id="b0f22-119">Ardından, aynı hizmet örneği içinde aynı anda hizmetin her iki sürümünü de uygulamak veya her biri API'nin bir sürümünü işleyen farklı örnekleri dağıtmak arasında karar verebilirsiniz.</span><span class="sxs-lookup"><span kitaplık)](https://github.com/jbogard/MediatR)bağımsız işleyicileri içine farklı uygulama sürümleri ayırmak için.</span><span class="sxs-lookup"><span data-stu-id="b0f22-120">A good approach for this is the [Mediator pattern](https://en.wikipedia.org/wiki/Mediator_pattern) (for example, [MediatR library](https://github.com/jbogard/MediatR)) to decouple the different implementation versions into independent handlers.</span></span>

<span data-ttu-id="b0f22-121">Son olarak, bir REST mimarisi kullanıyorsanız, [Hypermedia](https://www.infoq.com/articles/mark-baker-hypermedia) hizmetlerisürüm ve evolvable API'ler izin için en iyi çözümdür.</span><span class="sxs-lookup"><span data-stu-id="b0f22-121">Finally, if you're using a REST architecture, [Hypermedia](https://www.infoq.com/articles/mark-baker-hypermedia) is the best solution for versioning your services and allowing evolvable APIs.</span></span>
